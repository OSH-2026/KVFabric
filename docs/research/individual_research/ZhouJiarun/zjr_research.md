# ZhouJiarun Individual Research Report

## 个人调研主题

本阶段个人调研围绕三个方向展开：

1. `eBPF`
2. `协程调度器`
3. `KV Cache`

调研目标不是简单做概念罗列，而是从课程项目选题的角度，比较它们的系统属性、实现门槛、可验证性和最终落地价值，为后续大作业方向收敛提供依据。

## 一、eBPF 调研

### 1.1 基本概念

eBPF 可以理解为一种运行在内核受限执行环境中的可验证程序机制。用户可以将小型程序挂载到内核或用户态的特定事件点上，在不直接修改内核源码的前提下完成观测、过滤、统计、网络处理与安全策略执行。

在现代 Linux 系统中，eBPF 主要被广泛用于：

- 可观测性与性能分析
- 网络流量过滤与转发
- 安全监控与访问控制
- 内核事件追踪

### 1.2 为什么它有吸引力

从 OSH 课程视角看，eBPF 的吸引力非常强：

- 它天然具有“操作系统味道”，涉及内核、系统调用、调度、网络和安全。
- 它兼具理论和工程性，既能讲原理，也能做真实实验。
- 可展示性强，容易做 demo 和性能图表。

如果围绕 eBPF 选题，可以考虑的方向包括：

- 基于 eBPF 的系统调用追踪与性能分析
- 基于 eBPF 的网络观测或包过滤工具
- 面向特定服务的热点函数跟踪与延迟分析

### 1.3 面临的难点

eBPF 的门槛也非常明显：

- 强依赖 Linux 内核环境
- 对开发和调试环境要求较高
- 需要较强的内核、工具链与权限管理知识
- 很多问题会卡在环境配置和 verifier 约束上，而不是核心创新本身

对于课程大作业来说，这意味着：

- 如果团队对 Linux 内核和工具链不熟，前期成本会非常高
- 很多时间可能花在环境与权限问题上
- 项目容易做成“工具复刻”或“追踪脚本集合”，创新点不够聚焦

### 1.4 阶段判断

eBPF 是很强的系统方向，但它更适合作为“内核观测/网络/安全”类项目主线。如果目标是做一个完整、可量化、可对比的系统优化课题，它需要非常明确的切入问题，否则容易陷入“能观测但难形成完整研究故事”的局面。

## 二、协程调度器调研

### 2.1 基本概念

协程调度器本质上是在用户态管理大量轻量任务的执行顺序与切换逻辑。相比内核线程，它强调更低的切换成本和更高的并发组织能力，常见于高性能网络服务、语言运行时和异步框架中。

调研时重点关注了以下问题：

- 协程与线程的关系
- 用户态调度与内核调度的边界
- 调度策略如何影响吞吐与延迟
- 协程调度器如何与 I/O 多路复用配合

### 2.2 可能的项目方向

围绕协程调度器，可以延伸出不少课程项目主题：

- 一个简化版用户态协程运行时
- 带优先级或 work-stealing 的协程调度器
- 面向特定 workload 的协程调度策略优化
- 对比线程池与协程调度在高并发场景下的表现

### 2.3 优势

这个方向的优点在于：

- 非常符合操作系统与运行时系统课程语境
- 可以直接落到代码实现，不依赖特别重的外部环境
- 结构清晰，便于展示调度策略、上下文切换、任务队列等机制

### 2.4 局限

但协程调度器也存在几个现实问题：

- 相关课程项目很多，容易做得比较常规
- 如果只是“从零写一个协程库”，创新性不一定足够
- 想做得更深入时，需要补上网络栈、I/O 模型、负载模型等大量外围内容

也就是说，这个方向比较适合“实现型课程项目”，但如果目标是做出较强的系统优化故事，还需要找到一个更具体、更有研究空间的切入点。

### 2.5 阶段判断

协程调度器是一个实现风险较低、系统味道较足的备选方向。如果后续项目希望更多强调“运行时设计”和“调度策略”，它依然是不错的选择；但就当前团队想做的创新密度和应用场景而言，它还不够贴近现阶段的目标问题。

## 三、KV Cache 调研

### 3.1 基本概念

KV Cache 是大语言模型推理中的核心系统机制之一。对于自回归生成模型，在生成每个新 token 时，如果每一步都重新计算全部历史上下文，代价会非常高。KV Cache 的基本思想是缓存历史 token 在 attention 中对应的 `Key / Value` 表示，从而避免重复计算。

它直接关系到：

- 长上下文推理性能
- 高并发服务的显存压力
- decode 阶段吞吐
- prefix reuse 与共享上下文优化

### 3.2 为什么这个方向值得做

KV Cache 兼具“AI 系统热点问题”和“操作系统式资源管理问题”两种属性：

- 一方面，它是当前 LLM serving 框架中的关键瓶颈之一；
- 另一方面，它天然涉及分页、分配、共享、回收、驱逐、重算代价等系统问题。

这使它很适合作为 OSH 课程的大作业题目，因为可以把操作系统中的资源管理思想迁移到 LLM 推理系统中。

### 3.3 现阶段看到的关键问题

调研中最值得关注的几个问题包括：

- KV Cache 占用随上下文长度线性增长
- 高并发场景下 block 分配与回收会带来碎片与管理开销
- 现有 prefix caching 对“完全相同前缀”更友好，对局部分叉和模板化场景支持不足
- 分配、复用、驱逐往往分别处理，缺乏统一协同

这些问题背后既有性能约束，也有系统结构问题，因此研究空间很明确。

### 3.4 适合切入的方向

围绕 KV Cache，可以继续细分出若干方向：

- KV Cache 分页分配与碎片控制
- Prefix caching 与近似前缀复用
- 多请求共享上下文下的生命周期管理
- 驱逐策略与重算成本建模
- Copy-on-Write 分叉与增量复用

在多个候选方向中，最适合作为课程大作业主线的是：

> 面向 LLM 推理服务的统一 KV Cache 生命周期管理框架

这个方向既能体现系统设计，又能留下可量化的实验指标。

### 3.5 阶段判断

相比 eBPF 和协程调度器，KV Cache 更适合当前项目，主要原因有三点：

- 它的研究问题更聚焦，容易形成完整的问题定义、设计方案和实验故事。
- 它既有系统深度，也有现实应用背景，适合课程展示。
- 它允许在已有框架基础上做增量改造，开发风险相对更可控。

## 四、三者对比

| 维度 | eBPF | 协程调度器 | KV Cache |
|---|---|---|---|
| 系统属性 | 很强 | 很强 | 很强 |
| 开发环境门槛 | 高 | 中 | 中 |
| 创新点提炼难度 | 中到高 | 中 | 中 |
| 工程可落地性 | 中 | 高 | 高 |
| 可量化评测 | 中 | 高 | 高 |
| 与当前团队方向契合度 | 一般 | 较高 | 最高 |

从课程项目落地角度看：

- `eBPF` 更像一个“强系统、强环境依赖”的方向；
- `协程调度器` 更像一个“实现稳定、但需要额外创新点”的方向；
- `KV Cache` 则更像一个“系统性与现实价值兼顾”的方向。

## 五、最终结论

本阶段个人调研的结论是：

1. `eBPF` 很有系统深度，但环境依赖较强，且需要更明确的场景切入。
2. `协程调度器` 适合作为备选实现型课题，但主题相对常见，后续需要额外强化创新性。
3. `KV Cache` 最适合作为当前项目主线，特别是从“统一生命周期管理 + chunk 级复用 + CoW 分叉”这一方向继续深入。

因此，后续项目重点转向：

- 以 `vLLM` 为主要基座
- 从系统资源管理的角度研究 KV Cache
- 将分配、复用、驱逐和分叉统一到一个框架中

## 六、后续建议

- 继续阅读 vLLM 中与 block manager、scheduler、prefix cache 相关的代码路径
- 明确统一元数据结构与状态流转
- 先做 Python 控制面的原型，再决定是否需要下沉到底层实现
- 尽早设计 benchmark，避免后期只有实现、没有足够清晰的评测结果
