# “面向 LLM 推理服务的 KVCache 分配、复用与驱逐协同优化”项目中 vLLM 与 llama.cpp 适用性调研报告

## 摘要

在“面向 LLM 推理服务的 KVCache 分配、复用与驱逐协同优化”项目中，选择何种基础推理框架，将直接影响研究问题的可达性、实验设计的有效性以及后续原型实现的复杂度。当前开源生态中，vLLM 与 llama.cpp 分别代表了两种不同的系统路线：前者面向服务端高吞吐、高并发与 KVCache 精细治理，后者面向轻量部署、本地推理与广泛硬件适配。两者都支持大模型推理，但在系统目标、缓存组织方式、服务抽象层次以及可扩展研究空间上存在明显差异。

本项目关注的核心问题不是“让模型能够运行”本身，而是围绕 KVCache 的分配、共享复用与驱逐回收之间的耦合关系，构建统一生命周期管理框架，并进一步研究 prefix caching、共享 block 管理、copy-on-write、协同驱逐策略等机制。这类问题天然属于服务系统级优化，需要底层框架具备显式的 KV block 管理、缓存元数据维护、并发请求调度、前缀缓存命中与回收机制，以及足够清晰的系统层抽象。

从这一角度看，vLLM 与本项目的匹配度明显高于 llama.cpp。vLLM 已经具备 PagedAttention、Automatic Prefix Caching、block 化 KVCache 管理、引用关系维护与回收逻辑，项目能够在现有抽象之上直接开展生命周期建模与策略改进；而 llama.cpp 虽然更加轻量、易于本地运行、CPU 友好，但其设计重点仍然是“以尽量低门槛在多种硬件上高效执行模型”，并未将请求间共享复用、服务级缓存调度与 block 生命周期治理作为核心抽象对象。因此，若以 llama.cpp 为主框架推进本项目，将面临更高的系统补全成本，研究重点也容易从“协同优化”偏移到“先补基础设施”。

综合调研可以得出结论：对于“面向 LLM 推理服务的 KVCache 分配、复用与驱逐协同优化”这一项目，vLLM 更适合作为主要研究平台；llama.cpp 适合作为轻量级对照实现、边缘侧参考或 CPU 场景下的补充实验平台，但不宜作为该项目的主框架。

**关键词：** KVCache；vLLM；llama.cpp；LLM 推理服务；Prefix Caching；缓存驱逐；协同优化

## 1. 调研背景

随着大语言模型推理服务规模持续扩大，KVCache 已成为影响系统时延、吞吐率和显存利用率的关键资源之一。[1][4] 对于自回归生成模型而言，KVCache 用于保存历史 token 对应的 Key/Value 状态，以降低解码阶段的重复计算成本。然而，随着长上下文、多轮对话和高并发服务需求增长，KVCache 的空间占用与管理复杂度迅速上升，围绕其分配、复用和驱逐的优化问题，已经成为推理系统研究的重要方向。[1][4]

本项目以“面向 LLM 推理服务的 KVCache 分配、复用与驱逐协同优化”为核心目标，拟提出一种统一的 KVCache 生命周期管理框架，将分配、共享复用与驱逐回收纳入同一设计视角，并进一步探索近似前缀匹配、chunk 级共享、copy-on-write 和共享感知驱逐策略等机制。在这种研究目标下，基础推理框架的选择不只是一般意义上的工程部署问题，而是直接决定项目研究能否在合理工作量下展开的关键前置条件。

当前开源生态中，vLLM 与 llama.cpp 都是广泛使用的推理系统，但二者定位明显不同。为了明确哪一个更适合作为本项目的基础平台，有必要从系统设计目标、缓存组织方式、并发模型、共享支持、驱逐逻辑、代码改造空间和实验适配性等方面进行系统对比。

## 2. 项目对基础框架的需求分析

在选择基础框架之前，首先需要明确本项目对底层推理系统提出了哪些要求。

### 2.1 项目研究的本质不是“推理加速”，而是“缓存系统协同优化”

本项目的核心问题并不是单纯提高某次推理的执行速度，而是研究一个在服务场景中动态变化的缓存系统。具体来说，项目关注的是：

1. KV block 如何分配；
2. 不同请求如何共享已有前缀缓存；
3. 当请求发生局部分叉时如何管理共享关系；
4. 当显存紧张时哪些 block 应优先驱逐；
5. 被驱逐后的重算代价是否值得；
6. 分配、复用和驱逐之间如何形成协同机制。

因此，本项目真正需要的并不是一个“能跑模型”的引擎，而是一个已经显式暴露缓存管理问题、能够承载系统级策略实验的推理框架。

### 2.2 对基础框架的关键要求

结合研究目标，本项目理想的基础框架应具备以下能力：

- **显式的 KVCache 管理结构**，最好以 block/page 为基本单位；
- **可观测的缓存元数据**，例如引用计数、访问时间、所属请求和共享状态；
- **多请求服务调度能力**，便于研究共享和驱逐在真实服务中的作用；
- **已有的 prefix caching 或相近机制**，便于在其基础上增强；
- **存在驱逐/回收逻辑**，便于研究协同驱逐策略；
- **源码结构足够清晰**，可以定位 scheduler、cache manager、worker 等关键模块；
- **能够支撑实验对比**，包括命中率、延迟、吞吐、显存利用率和重算开销等指标。

从这些要求可以看出，本项目天然更适合一个“面向服务系统设计”的推理框架，而不只是一个“通用本地推理程序”。

## 3. vLLM 的特点及与项目的匹配度

### 3.1 vLLM 的基本定位

vLLM 是一个面向大规模服务场景设计的 LLM 推理框架，其代表性机制是 PagedAttention，并以此为基础追求高吞吐、低碎片和灵活共享。[1] 在公开设计文档中，vLLM 还进一步给出了 Automatic Prefix Caching 的数据结构、分配流程、回收逻辑与命中机制。[2] 从系统设计目标来看，vLLM 天然面向“服务场景中的缓存与调度问题”，这一点与本项目高度一致。

### 3.2 vLLM 对 KVCache 的显式管理能力

vLLM 相比许多普通推理框架的最大特点，是它并没有把 KVCache 仅仅作为某个算子的内部状态，而是将其提升为服务系统核心资源来管理。具体体现在：

1. **KVCache 被按 block/page 切分管理**  
   PagedAttention 将 KV cache 组织为可分页、可离散管理的 block，从而避免传统连续分配带来的显著碎片浪费。[1][3]

2. **系统显式维护 block 元数据**  
   vLLM 的 Automatic Prefix Caching 设计文档明确给出了 KVCacheBlock、block pool、free block queue、cache blocks 与 request blocks 等结构，这为研究 block 生命周期、共享状态和驱逐策略提供了直接切入点。[2]

3. **存在请求间前缀缓存复用机制**  
   vLLM 已内置 Automatic Prefix Caching，说明系统不仅管理“每个请求自己的缓存”，还允许不同请求围绕公共前缀进行复用。[2]

4. **存在回收与驱逐入口**  
   一旦显存紧张，系统需要决定哪些 block 继续保留，哪些 block 回收，这正是本项目关注的核心问题之一。[2]

换言之，vLLM 已经把本项目想研究的很多问题，以具体系统机制的形式摆在了研究者面前。

### 3.3 vLLM 与本项目创新点的直接契合

本项目的几个关键创新方向，与 vLLM 的现有机制之间存在很强的自然衔接关系：

#### 3.3.1 统一生命周期管理框架

vLLM 已经存在 block 分配、prefix reuse、引用关系和回收逻辑，因此非常适合进一步抽象为统一生命周期模型，而不需要从零设计对象体系。[1][2]

#### 3.3.2 协同驱逐策略

在 vLLM 当前结构中，block 已是可管理对象，且系统保留了共享与回收所需的关键状态。这使项目能够在现有元数据之上引入：

- 复用潜力；
- 重算代价；
- 共享深度；
- 前缀层级位置；

从而实现生命周期感知的驱逐优先级。

#### 3.3.3 局部分叉与 copy-on-write

由于 vLLM 已以 block 为单位组织缓存，因此更容易扩展出“共享后逐步分叉”的管理策略，例如：

- 保留共享前缀 block；
- 为分叉部分增量分配新 block；
- 对共享路径进行结构化管理；
- 引入 copy-on-write 或延迟复制。

#### 3.3.4 近似前缀或 chunk 级共享

虽然 vLLM 当前更偏向 exact-match 的前缀复用，但其缓存共享与管理机制已经存在，这意味着项目可以在既有实现上增强，而不是先从零补一套共享缓存系统。[2]

### 3.4 vLLM 的不足与挑战

尽管 vLLM 很适合本项目，但也并非没有问题：

1. **源码复杂度较高**  
   涉及 engine、scheduler、worker、model runner、cache manager 等多个模块，学习成本较高。

2. **依赖环境偏重**  
   GPU 依赖明显，对实验环境要求较高。

3. **系统改动门槛较大**  
   因为它已经是一个成熟服务框架，修改核心缓存路径需要较强的源码理解能力。

不过，这些挑战更多属于实现复杂度问题，而不是研究方向错位问题。从“能否承载本项目的核心问题”来看，vLLM 仍然是更合适的选择。

## 4. llama.cpp 的特点及与项目的匹配度

### 4.1 llama.cpp 的基本定位

llama.cpp 在其公开说明中将自己定位为“LLM inference in C/C++”，强调 minimal setup、广泛硬件适配、本地与云端可运行以及量化友好。[5] 它同时提供 CLI、轻量级 OpenAI 兼容 HTTP server，并支持多用户与 parallel decoding。[5] 这说明 llama.cpp 并非不能用于服务化部署，但其主目标仍然是“以尽量低门槛让模型在多种硬件上跑起来”，而不是围绕服务级缓存治理建立系统抽象。

### 4.2 llama.cpp 在 KVCache 层面的特点

llama.cpp 当然也有 KVCache，因为自回归生成本身离不开历史状态缓存。但从整体设计叙事和工程组织方式来看，它的 KVCache 更偏向：

- 为单次推理或单会话执行服务；
- 优先保障基础推理性能与可移植性；
- 以推理引擎视角管理缓存；
- 不强调多请求共享、服务级缓存调度和统一生命周期治理。

换言之，在 llama.cpp 中，KVCache 更多是“模型运行时的一部分”，而不像 vLLM 那样被提升为服务系统中的独立资源管理对象。

### 4.3 llama.cpp 对本项目的局限

如果将 llama.cpp 作为本项目主框架，会遇到以下问题：

#### 4.3.1 缺少成熟的服务级 prefix caching 抽象

本项目希望研究请求间共享复用，而 llama.cpp 的主设计目标并不在此。若要研究前缀共享，往往需要额外补充：

- 请求管理层；
- 缓存共享判定机制；
- block 元数据维护；
- 引用计数与共享拓扑；
- 驱逐策略体系。

这会导致项目大量时间消耗在“搭框架基础设施”上，而不是研究协同优化本身。

#### 4.3.2 缺少显式 block 生命周期管理结构

如果底层缓存尚未被提升为 page/block 级可管理对象，那么研究“分配、复用与驱逐的统一生命周期”就会失去天然的对象基础。项目需要先把 llama.cpp 改造成更像服务系统缓存管理器的形态，这会显著增加工作量。

#### 4.3.3 服务调度与缓存治理并非主线抽象

虽然 llama.cpp 提供 lightweight server 和 parallel decoding 支持，[5] 但其主要优势仍然集中在轻量化、量化支持与广泛硬件后端，而不是围绕高并发推理服务的缓存调度问题建立研究入口。因此，若要模拟本项目关注的真实服务场景，仍需额外构建上层控制逻辑。

### 4.4 llama.cpp 的优势与可利用之处

尽管 llama.cpp 不适合作为本项目主框架，但它仍然具有明显价值：

1. **轻量、易运行**  
   适合做小规模 CPU 对照实验。

2. **工程路径较直接**  
   有助于理解基础推理流程和 KVCache 的原始运行路径。

3. **适合边缘侧或本地侧场景讨论**  
   如果项目后期想讨论“协同优化是否能下沉到轻量推理引擎”，llama.cpp 可以作为参考平台。

4. **适合做最小化概念验证**  
   若只想快速验证某类共享思想在单机环境下是否存在潜在收益，llama.cpp 可以作为低成本试验载体。

因此，llama.cpp 更适合作为“补充参考平台”，而不是主研究载体。

## 5. 从项目目标出发的对比分析

### 5.1 总览对比表

| 比较维度 | vLLM | llama.cpp | 对本项目的含义 |
| --- | --- | --- | --- |
| 系统定位 | 面向服务端高吞吐与动态 KVCache 管理[1] | 面向最小依赖、广泛硬件与本地高效推理[5] | 项目问题域明显更接近前者 |
| KVCache 抽象粒度 | block/page 化，且存在独立 manager 与元数据[1][2] | 以执行上下文为主，缺少成熟共享治理抽象 | vLLM 更容易承载生命周期研究 |
| 请求间共享 | 已有 Automatic Prefix Caching[2] | 需要自行补充共享判定与管理机制 | llama.cpp 的前置工程量更大 |
| 调度与服务能力 | 请求调度、engine/worker 分层清晰 | 有轻量 server，但服务调度不是核心抽象[5] | vLLM 更适合做服务级实验 |
| 驱逐与回收入口 | 已有 free queue 与回收流程[2] | 需要自行系统化设计 | vLLM 更适合开展协同驱逐 |
| 部署与上手成本 | 学习成本高、GPU 依赖重 | 轻量、易编译、CPU 友好[5] | llama.cpp 适合辅助验证 |
| 更适合承担的角色 | 主研究平台 | 轻量对照与补充实验平台 | 推荐采用“双平台、主辅分工”策略 |

### 5.2 研究问题可达性

- **vLLM**：研究问题已部分以系统机制形式存在，可直接深入；
- **llama.cpp**：需要先补齐服务化缓存管理基础设施，再谈核心研究问题。

因此，从“更快进入核心研究问题”的角度，vLLM 更优。

### 5.3 与协同优化主题的一致性

本项目强调的是 allocation、reuse/sharing、eviction 三者之间的耦合关系。vLLM 已经具备这三个环节的系统化表达；而 llama.cpp 更多只是具备“缓存存在”这一事实，缺少系统级协同管理抽象。

因此，从主题一致性角度，vLLM 更优。

### 5.4 实验设计可行性

本项目拟比较的核心指标包括：

- 命中率；
- 显存利用率；
- 重算代价；
- 请求延迟；
- 吞吐率。

这些指标与服务调度、共享路径和显存管理紧密相关。vLLM 天然更适合构造这类实验；llama.cpp 则更适合单机延迟、量化收益或 CPU 性能测试。

因此，从实验指标适配性角度，vLLM 更优。

### 5.5 实现成本与学习成本

- **vLLM**：学习成本高，但改动属于“在已有合适框架中做针对性增强”；
- **llama.cpp**：上手门槛较低，但若要支撑本项目目标，后续重构成本会显著上升。

因此，短期看 llama.cpp 更容易入门；但从完成项目目标的总成本看，vLLM 反而更经济。

### 5.6 研究成果的说服力

如果项目最终在 vLLM 上实现了：

- 生命周期建模；
- 共享感知驱逐；
- chunk 级共享；
- copy-on-write 分叉管理；

那么成果与“LLM 推理服务系统优化”之间的关联会非常直接，学术和工程说服力都更强。相反，若在 llama.cpp 上实现类似机制，评审者很可能会质疑：这些机制究竟是在解决典型服务系统的缓存痛点，还是在补足一个原本并不以服务级缓存治理为核心的框架。

因此，从成果表达与研究可信度角度，vLLM 更优。

## 6. 建议的使用策略：主框架与辅助框架并存

虽然本报告结论是 vLLM 更适合作为主平台，但这并不意味着 llama.cpp 完全没有位置。更务实的项目策略，应当是“主框架承担研究主体，辅助框架承担验证与对照”。

### 6.1 以 vLLM 作为主研究平台

vLLM 应承担以下主要工作：

- 源码分析与关键路径梳理；
- KVCache 生命周期建模；
- block 级共享与驱逐策略设计；
- 主体实验与性能对比；
- 最终原型实现。

更具体地说，项目可先在 exact-prefix 共享和现有回收路径之上建立 baseline，再逐步引入共享感知驱逐、局部分叉管理和 chunk 级扩展策略。[1][2]

### 6.2 以 llama.cpp 作为辅助参考平台

llama.cpp 更适合承担以下任务：

- 小规模 CPU 环境快速验证；
- 对比不同框架中 KVCache 的基础组织差异；
- 讨论本项目方案是否具有向轻量推理框架迁移的可能性；
- 在资源受限环境中做概念性演示。

这类使用方式既发挥了 llama.cpp 轻量、易部署的优势，又避免其成为主框架后带来的系统补全负担。

### 6.3 建议的实施路径

为了保证项目推进节奏清晰，建议采用分阶段实施策略：

1. **阶段一：读通现有缓存路径**  
   在 vLLM 中梳理 allocation、prefix reuse、free/eviction 的关键调用链，形成最小生命周期状态机。

2. **阶段二：建立可复现实验基线**  
   固定模型、上下文长度、并发度和请求模式，记录命中率、显存利用率、吞吐率与重算开销，形成对照基线。

3. **阶段三：引入协同优化策略**  
   优先实现共享感知驱逐，再逐步扩展到 copy-on-write、chunk 级共享和局部分叉管理。

4. **阶段四：做轻量对照与迁移讨论**  
   在 llama.cpp 或其他轻量环境中做概念性验证，用于说明方案的可迁移性边界，而不把主体创新建立在轻量框架补基础设施之上。

### 6.4 主要风险与应对

在项目实施过程中，还需提前考虑以下风险：

1. **源码复杂度高，调试成本大**  
   应对方式是优先收敛修改范围，只围绕 cache manager、scheduler 和关键路径插桩，而不是大面积改动框架。

2. **实验结果容易受 workload 影响**  
   应对方式是固定请求模板、上下文长度与并发度，避免仅凭少量偶然样例得出结论。

3. **方案可能过度绑定 vLLM 实现细节**  
   应对方式是在设计时保留抽象层，把策略表述为“生命周期感知缓存管理”而不是“某个具体类的局部技巧”。

4. **硬件资源有限影响迭代速度**  
   应对方式是先使用较小模型和离线回放 workload 完成机制验证，再逐步扩展到更大规模实验。

## 7. 结论

围绕“面向 LLM 推理服务的 KVCache 分配、复用与驱逐协同优化”这一项目目标，vLLM 与 llama.cpp 虽然都属于 LLM 推理框架，但二者面向的问题域并不相同。

vLLM 是一个以服务系统为中心构建的框架，其核心优势恰恰体现在：

- block/page 化 KVCache 管理；
- Automatic Prefix Caching；
- 多请求调度；
- 缓存共享与回收的系统化抽象。[1][2]

这些机制与本项目拟研究的“统一生命周期视角”“共享复用与驱逐联动”“协同驱逐策略”“局部分叉与增量管理”等方向高度一致，因此 vLLM 非常适合作为本项目的主研究平台。

相比之下，llama.cpp 的优势在于轻量化、低依赖和本地部署友好，[5] 但它更偏向单机推理引擎与通用轻量执行框架，而不是围绕并发服务与缓存协同治理构建研究入口。如果将 llama.cpp 作为主框架，项目将不得不先补足大量服务级缓存管理基础设施，从而分散研究重点、增加工作量，并削弱最终成果与“推理服务系统优化”之间的直接对应关系。

因此，本报告给出的结论是：

1. **vLLM 更适合作为本项目的主框架；**
2. **llama.cpp 可作为补充参考平台或轻量级对照实现；**
3. **若项目目标聚焦于 KVCache 的分配、复用与驱逐协同优化，则应优先在 vLLM 上开展源码分析、策略设计和原型实现。**

进一步说，最合理的工程策略并不是“二选一”，而是采用“vLLM 为主、llama.cpp 为辅”的主辅分工模式：研究与原型在 vLLM 上完成，轻量验证与迁移讨论在 llama.cpp 上展开。这一组合能够在研究深度、实现成本与成果说服力之间取得更好的平衡。
