# 面向自研 LoongArch CPU 的应用承载型轻量内核设计与实现调研报告

## 摘要

随着国产处理器、自主指令系统和信创软硬件生态的持续发展，面向自研 CPU 的基础软件栈建设正在从“能运行”走向“可承载、可扩展、可工程化”。在这一背景下，围绕自研 LoongArch CPU 设计一个既具有轻量特征、又能够承载上层应用运行的内核系统，具有明确的研究价值和工程意义。与传统通用操作系统相比，轻量内核更强调小体积、低资源占用、快速启动、易裁剪和硬件适配灵活；而与纯 RTOS 或教学内核相比，应用承载型内核又必须具备进程/线程管理、虚拟内存、异常中断、系统调用、基础文件系统与驱动框架等更完整的系统能力，从而支持较为真实的软件运行场景。

本课题的重要初衷之一，是为龙芯杯方向的自研 LoongArch CPU 作品提供一个可用于软硬件联调的系统软件平台。换言之，该内核不仅是一个独立的操作系统实现课题，也被期望承担启动验证、异常中断联调、用户程序装载验证和作品展示支撑等实际用途。

LoongArch 作为国产自主指令系统，已经进入 Linux 主线内核文档与软件生态，具备较好的体系结构公开资料和基础软件支持条件。Linux 内核文档已经给出了 LoongArch 的寄存器、基础指令、虚拟内存、启动流程与 IRQ 模型等体系结构信息；同时，公开报道也表明 LoongArch32 在嵌入式、教学与应用级处理器方向仍具有现实意义。[1][4] 这为面向自研 LoongArch CPU 的轻量内核设计提供了较清晰的体系结构基础。

本报告围绕“面向自研 LoongArch CPU 的应用承载型轻量内核设计与实现”这一方案进行系统调研。报告首先分析课题背景、研究意义和目标场景；其次梳理 LoongArch 架构特点以及轻量内核、微内核、宏内核、RTOS 等相关技术路线；随后对本方案的总体设计思路进行展开，重点讨论启动链路、异常中断、进程线程管理、内存管理、系统调用、驱动与文件系统、用户态应用承载等关键模块；在此基础上进一步分析该方案的技术可行性、实现路径、风险点和实验验证方法。最后给出项目预期成果与后续工作建议。

综合分析可知，面向自研 LoongArch CPU 开发应用承载型轻量内核是可行的，且具有较好的分阶段推进空间：前期可先实现最小可用内核，完成启动、串口、时钟、中断、任务调度与基本内存管理；中期进一步补齐用户态、系统调用、ELF 加载、虚拟内存和基础文件系统；后期则围绕应用兼容性、性能优化与工程化展开扩展。该方案适合作为龙芯杯类竞赛项目、课程设计课题或后续科研探索的基础平台。与此同时，也应看到其核心风险在于硬件研制进度与课程周期可能不一致：若目标 CPU 在课程结束前尚未达到可稳定联调状态，则项目需要转向仿真器、FPGA 原型或参考实现完成阶段性验证。

**关键词：** LoongArch；自研 CPU；轻量内核；应用承载；操作系统；系统软件；国产架构

## 1. 课题背景与研究意义

### 1.1 背景

在处理器自主可控和基础软件自主演进的背景下，围绕国产 CPU 架构构建完整的软件栈已成为体系结构、编译系统和操作系统方向的重要任务。处理器并不仅仅是一个能够执行指令的硬件实体，只有当其拥有可启动、可调试、可加载程序、可处理中断和可运行应用的系统软件环境时，才真正具备“平台”意义。因此，面向自研 CPU 的内核设计与实现，不仅是操作系统教学与实验的延续，更是芯片—系统软件协同设计的重要一环。

从项目来源看，本课题并非纯粹的纸面方案，其一个直接目标就是与龙芯杯方向的自研 LoongArch CPU 作品进行联调。也就是说，课题输出不只是“写出一个内核原型”，还希望在课程或竞赛周期内承担处理器启动后的软件承载、异常路径验证、应用加载演示和软硬件协同调试等职责。

对于 LoongArch 而言，其作为自主指令系统，已形成一定程度的软件生态基础。Linux 主线内核文档已经将 LoongArch 作为正式架构加以描述，内容涵盖寄存器、基础指令、虚拟内存、内核启动和 IRQ 模型等。[1] 这意味着 LoongArch 相关系统软件开发不再是完全“黑盒”状态，而是具备了较稳定的架构参考。

### 1.2 为什么是“应用承载型轻量内核”

单纯的裸机程序或极简教学内核通常只能完成少量功能验证，例如串口输出、异常陷入或简单任务切换；而通用 Linux 又过于复杂，移植成本高、调试链路长、硬件依赖多，不适合作为面向自研 CPU 的首个完整软件平台。相比之下，“应用承载型轻量内核”位于两者之间：

- 它比裸机或极简内核更完整，能够加载并运行用户态程序；
- 它比完整 Linux 更小、更易控，适合作为自研 CPU 的早期验证平台；
- 它既能用于硬件 bring-up，也能支持上层基础应用与运行时环境；
- 它适合作为课程、竞赛和研究中的系统级实验平台。

因此，该课题的核心并不是做一个“最小功能演示程序”，而是设计一个在资源可控前提下具备真实系统能力的轻量内核。

### 1.3 研究意义

该方案的意义主要体现在以下几个方面：

1. **服务于自研 CPU 验证与 bring-up。**  
   一个轻量但完整的内核可以用于验证异常、中断、TLB、时钟、中断控制器、串口、内存映射等关键硬件机制。

2. **构建自主软硬件协同平台。**  
   通过面向 LoongArch CPU 自主设计内核，可形成从 ISA、启动代码、ABI、系统调用到应用运行的完整链路。

3. **降低系统软件实验门槛。**  
   相比直接移植 Linux，一个轻量内核更容易理解和控制，适合作为教学、竞赛与研究入门平台。

4. **支持后续生态扩展。**  
   若内核具备基础应用承载能力，则后续可逐步扩展为支持文件系统、网络、Shell、工具链甚至类 POSIX 子集的平台。

5. **服务于龙芯杯作品联调与展示。**  
   若自研 CPU 能在项目周期内达到可运行状态，该内核可直接作为作品的软件侧支撑平台，用于展示启动链路、异常处理、任务调度、用户程序装载和基础系统调用等能力。

## 2. LoongArch 架构与相关技术基础调研

### 2.1 LoongArch 架构基础

LoongArch 是国产自主指令系统。Linux 内核文档对 LoongArch 的架构信息已有较完整描述，包括通用寄存器、浮点寄存器、LSX/LASX 向量扩展、基础整数指令、访存指令、原子操作、CSR/IOCSR 等特权相关指令，以及虚拟内存和内核启动接口。[1]

除 Linux 文档外，龙芯官方还公开了 LoongArch 指令集与 ELF psABI 文档，可为寄存器约定、异常返回路径、二进制接口以及程序装载实现提供更底层的规范依据。[6]

从内核设计角度看，以下几个方面尤其重要：

- **特权级与异常机制**：决定内核态/用户态隔离方式；
- **CSR 与异常返回机制**：决定陷入与恢复现场逻辑；
- **虚拟内存/TLB 机制**：决定是否支持用户进程地址空间隔离；
- **中断模型**：决定时钟、IPI、外设中断的接入方式；
- **原子与同步指令**：影响锁、调度和 SMP 扩展能力。

因此，LoongArch 为实现一个应用承载型轻量内核提供了必要的架构基础，而不是仅适合裸机开发。

### 2.2 LoongArch 在 Linux 生态中的支撑

LoongArch 已进入 Linux 内核主线文档体系。Linux 文档中不仅有架构介绍，还有启动 Linux/LoongArch、内核镜像头、IRQ 模型等内容。[2][3]  
这说明：

1. 架构接口和启动约定已经较明确；
2. 存在可参考的异常、中断和虚拟内存实现思路；
3. 相关工具链、ABI 与软件栈已有一定基础。

此外，公开报道显示，LoongArch32 仍在嵌入式和教学方向具有现实使用价值，并已有 openLA500、openLA1000 等开源参考实现被提及。[4]  
这对竞赛和实验类项目尤其有意义：说明面向 LoongArch 的系统软件不仅停留在高端服务器方向，也适用于轻量系统与教学平台。

### 2.3 轻量内核的常见技术路线

面向自研 CPU 的轻量内核通常有以下几种路线：

#### 2.3.1 宏内核裁剪路线

即保留内核的核心管理能力，在一个统一内核地址空间中实现进程、线程、内存管理、文件系统和基础驱动。这一路线实现简单、性能直接、调试方便，适合作为竞赛项目和首个自研 CPU 内核平台。

#### 2.3.2 微内核路线

即将进程、文件系统、驱动等尽可能下沉到用户态，以消息通信为主。这种设计理论上结构更清晰、隔离更强，但对 IPC、用户态服务与调试要求高，实现复杂度较大，不适合作为项目初版首选。

#### 2.3.3 RTOS 扩展路线

从 LiteOS、FreeRTOS 一类系统出发，逐步增加 MMU、用户态和应用承载能力。这种方式能较快获得可运行内核，但如果原始系统并非为进程隔离和应用承载设计，后续扩展成本会比较高。

综合看，若课题目标是“应用承载型轻量内核”，则更适合采用**轻量宏内核 + 分层模块化设计**路线。

### 2.4 相关移植与国产平台经验启发

已有研究曾完成 OpenHarmony LiteOS-M 在龙芯 MIPS 平台上的移植与适配，并围绕 HAL、启动、串口驱动、内核裁剪和工具链构建展开设计与验证。相关实验表明，这类轻量系统可以在龙芯控制终端平台上稳定运行。[5]  
虽然该工作面向的是 MIPS 架构与 OpenHarmony，而非 LoongArch 自研 CPU，但其方法论具有参考价值：

- 先完成启动和串口；
- 再做 HAL 和中断时钟；
- 然后构建任务管理与裁剪内核；
- 最后逐步补齐驱动与验证链路。

这说明本课题若采用“分阶段推进”的策略，是有现实可行性的。

## 3. 课题目标与需求分析

### 3.1 总体目标

设计并实现一个面向自研 LoongArch CPU 的应用承载型轻量内核，使其具备以下能力：

- 可在目标 CPU 或仿真平台上成功启动；
- 支持基本异常与中断处理；
- 支持任务调度与线程切换；
- 支持用户态应用装载与执行；
- 支持基本的系统调用机制；
- 支持基础内存管理与可选虚拟内存；
- 支持最基本的设备驱动和调试输出；
- 可运行简单用户程序或小型运行时环境。

### 3.2 功能需求

根据“应用承载型”定位，内核至少应包含以下模块：

1. Boot/Start-up 启动模块；
2. 异常与中断子系统；
3. 时钟与调度器；
4. 线程/进程管理；
5. 地址空间与内存管理；
6. 用户态切换与系统调用；
7. ELF 或简化程序装载器；
8. 串口控制台；
9. 基础驱动与平台抽象层；
10. 可选的文件系统或 ramfs。

### 3.3 非功能需求

除功能外，课题还应满足若干非功能性目标：

- **轻量化**：代码规模和运行资源占用可控；
- **可移植**：与具体 SoC 解耦，具备平台抽象层；
- **可扩展**：后续可增加文件系统、网络、SMP 等能力；
- **可调试**：串口日志、异常现场打印、简单调试工具可用；
- **可验证**：能通过一组基础测试验证稳定性和正确性。

### 3.4 约束条件

本课题面临的主要约束包括：

- 自研 CPU 文档可能不完整；
- 外设可能较少，初期主要依赖串口和时钟；
- 若 MMU/TLB 实现不完善，用户态隔离功能实现会受限；
- 工具链与仿真平台支持可能需要自行适配。

因此，方案设计必须兼顾“目标完整性”和“阶段可落地性”。

## 4. 总体方案设计

### 4.1 设计原则

本方案遵循以下原则：

1. **先跑起来，再逐步增强**；
2. **内核最小闭环优先**；
3. **平台相关与平台无关代码分离**；
4. **优先保证可调试性和可验证性**；
5. **保留向更完整 OS 演化的接口。**

### 4.2 分层架构

建议将系统划分为如下层次：

- **硬件平台层**：CPU、CSR、时钟、中断控制器、串口、内存映射；
- **架构相关层**：启动代码、上下文切换、异常入口、TLB 操作；
- **平台抽象层 HAL**：封装平台差异，向内核提供统一接口；
- **内核核心层**：调度、任务管理、内存管理、同步原语；
- **系统服务层**：系统调用、程序装载、VFS/ramfs；
- **用户应用层**：Shell、测试程序、示例应用。

这种分层能保证系统既适合早期 bring-up，也方便后期增强。

### 4.3 推荐的实现阶段

为了控制项目风险，每个阶段都应以“可启动、可验证、可观测”为退出条件，即不仅要求功能存在，还要求能够通过串口日志、测试程序或回归脚本证明该阶段目标已经稳定达成。

#### 第一阶段：最小可运行内核

- Boot 汇编入口
- BSS 清零与栈初始化
- 串口输出
- 异常向量建立
- 时钟中断
- 简单线程与调度
- 内核态测试任务运行

#### 第二阶段：用户态与应用承载

- 用户态上下文
- 系统调用入口
- 地址空间管理
- 简化 ELF 装载
- 用户程序执行
- 简单 Shell 或用户测试程序

#### 第三阶段：系统增强

- 页式虚拟内存
- 文件系统/ramfs
- 更完整驱动框架
- 进程间通信
- SMP 或多核支持
- 更丰富的调试与性能统计

## 5. 关键模块设计分析

### 5.1 启动与引导模块

内核启动链路是整个系统的第一步。对于自研 LoongArch CPU，需要解决以下问题：

- 复位后 PC 位置与启动 ROM 约定；
- 早期栈建立；
- BSS 清零与数据段重定位；
- 异常向量设置；
- MMU/TLB 初始状态；
- 跳转到 C 语言内核入口。

若早期阶段硬件环境不复杂，可采用“BootROM/Loader + Kernel Entry”两段式结构。BootROM 或简单 Loader 负责把内核映像加载到指定物理地址，随后跳入内核入口。

### 5.2 异常与中断机制

一个可承载应用的内核必须正确处理以下事件：

- 同步异常：非法指令、地址错、系统调用、页异常等；
- 异步中断：时钟、串口、外设中断；
- 用户态到内核态陷入与返回。

LoongArch 架构相关文档已给出基础特权和相关指令体系，可为异常入口设计提供参考。[1]  
在实现上，建议：

- 统一异常入口；
- 保存通用寄存器与关键 CSR；
- 区分异常类型并分发；
- 在系统调用与中断返回时恢复上下文。

### 5.3 线程、进程与调度

轻量内核的调度器不必一开始就非常复杂，但必须可靠。建议采用：

- 内核线程 + 用户进程双层模型；
- 初期使用简单优先级或时间片轮转；
- 使用定时器驱动抢占调度；
- 为每个任务维护内核栈与上下文块。

在“应用承载型”目标下，线程与进程的区别应明确：

- 线程共享地址空间；
- 进程具备独立用户地址空间与资源描述；
- 初期也可先实现单地址空间用户任务，再逐步演化到完整进程模型。

### 5.4 内存管理

内存管理决定了系统能否从“教学 demo”走向“真实承载”。建议分阶段实现：

#### 阶段一：物理内存管理

- 伙伴系统或位图分配器；
- 页框分配接口；
- 内核堆分配器（slab 或简化 kmalloc）。

#### 阶段二：虚拟内存管理

- 内核页表建立；
- 用户空间地址布局；
- TLB 映射维护；
- 缺页异常处理。

如果目标 CPU 初期不具备完整 MMU 条件，也可先实现**物理内存 + 软件隔离 + 简化装载**，再向虚拟内存扩展。

### 5.5 系统调用与用户态支持

这是“应用承载型”与普通轻量内核最关键的分水岭。系统必须支持：

- 用户态程序进入内核；
- 参数传递与返回值规范；
- 基础系统调用集合。

建议首批系统调用包括：

- `write/read`（至少支持串口/控制台）
- `exit`
- `yield`
- `getpid`
- `fork/exec`（可后续实现）
- `mmap/brk`（视内存管理进度决定）

如果初期不实现 POSIX 完整接口，也应保留兼容扩展空间。

### 5.6 程序装载与应用运行

为了真正承载应用，内核需要一个最基础的装载器。建议：

- 支持简化 ELF 格式；
- 能把代码、数据段映射到用户地址空间；
- 初始化用户栈；
- 建立用户入口并切换执行。

若初期实现压力较大，也可以先采用“固定格式用户镜像 + 简化加载协议”，在系统成熟后再过渡到 ELF。

### 5.7 驱动与文件系统

应用承载不一定要求一开始就有完整块设备文件系统，但至少需要：

- 串口驱动；
- 定时器驱动；
- 中断控制器驱动；
- 简化的内存文件系统（ramfs）或内置程序镜像机制。

中期可考虑增加：

- VFS 抽象；
- 简单 inode/dentry 结构；
- 只读文件系统；
- 块设备模拟接口。

## 6. 技术可行性分析

### 6.1 架构可行性

LoongArch 已具备较明确的架构资料与 Linux 生态文档支撑，说明其异常、中断、寄存器与虚拟内存机制足以支持完整内核开发。[1]  
因此，从 ISA 与特权机制角度看，实现一个轻量应用承载型内核是可行的。

### 6.2 工程可行性

相比移植完整 Linux，自己实现一个轻量内核的优势在于：

- 模块边界清晰；
- 不依赖复杂驱动生态；
- 适合逐步 bring-up；
- 便于针对自研 CPU 做定制优化。

只要具备基本的串口、时钟和内存接口，内核就能逐步构建起来。

### 6.3 平台可行性

若目标平台支持以下条件，则可显著降低实现难度：

- 可用的交叉编译工具链；
- 仿真环境或 FPGA 验证平台；
- 串口输出；
- 周期性定时器；
- 基础中断控制逻辑；
- 足够的内存空间。

这些条件在大多数竞赛或实验平台中一般都能满足。

如果目标硬件板卡尚未完全稳定，也可以先采用模拟器、指令级仿真环境或 FPGA 原型平台验证启动、异常、中断和上下文切换等关键路径，再逐步迁移到最终硬件平台，以降低早期调试成本。

这也意味着，本课题对“最终自研 CPU 是否在课程周期内完全就绪”并非绝对强依赖。即便硬件作品在学期结束前尚未完成上板、时序收敛或稳定联调，软件部分仍可借助替代验证平台形成阶段性成果，并为后续真正上板后的联调预先打通关键路径。

### 6.4 分阶段实现可行性

本方案最大优势在于可分阶段推进，不必一次完成所有功能。  
即使最终未完全实现完整进程隔离和复杂文件系统，也可以通过：

- 启动成功；
- 中断正常；
- 调度可用；
- 用户程序可运行；

来证明方案主体成立。

## 7. 关键难点与风险分析

### 7.1 文档与硬件细节不足

对于自研 CPU，最大风险之一往往不是软件本身，而是硬件接口文档不足。  
例如：

- 异常号定义不清晰；
- TLB 行为与预期不一致；
- 中断控制器时序不稳定；
- 定时器寄存器描述不完整。

这会直接影响内核 bring-up 效率。

**应对办法：**

- 建立最小硬件验证程序；
- 用串口打印与寄存器 dump 辅助调试；
- 先做裸机验证，再进内核。

### 7.2 用户态承载复杂度高

“能切线程”和“能跑用户应用”不是一个量级的问题。  
用户态引入后，必须处理：

- 用户/内核栈切换；
- 系统调用 ABI；
- 地址空间隔离；
- 异常恢复；
- 程序装载与返回路径。

这是本课题最核心的技术难点。

### 7.3 虚拟内存与 TLB 调试困难

如果方案需要支持 MMU/TLB，则调试会显著复杂。  
一旦页表建立、TLB 刷新或异常处理路径有误，系统可能直接失去可观测性。

**应对办法：**

- 先做恒等映射；
- 先跑单地址空间用户程序；
- 再引入完整页表和用户空间隔离。

### 7.4 应用生态有限

轻量内核初期很难直接承载复杂用户程序，因此“应用承载”应合理定义。  
项目初期的“应用”更适合指：

- 自编译的用户测试程序；
- 简单交互式 Shell；
- 小型 benchmark；
- 演示型组件。

不要一开始就追求大规模软件兼容。

### 7.5 硬件研制进度与课程周期错位

对于本课题而言，一个非常现实的风险是：项目的一大初衷虽然是与龙芯杯方向的自研 CPU 作品联调，但处理器本身未必能在课程结束前完成到“可稳定运行并可持续联调”的阶段。风险可能表现为：

- CPU 核功能尚未完全闭合；
- 仿真能跑但 FPGA 上板效果不稳定；
- 时钟、中断、存储控制器等关键硬件路径仍存在问题；
- 处理器可启动但不足以支撑用户态程序或复杂异常路径验证。

这类风险会直接影响内核在最终目标硬件上的验证深度，进而影响“应用承载型”目标的完整呈现。

**应对办法：**

- 将项目目标拆分为“内核实现完成度”和“目标硬件联调完成度”两个相对独立的维度；
- 优先在模拟器、参考核或 FPGA 原型上验证启动、异常、中断、调度和装载链路；
- 为真正的目标 CPU 预留清晰的 HAL 与架构适配接口，降低后续迁移成本；
- 在课程周期内以可验证的软件闭环为主，硬件联调部分作为加分项和后续延伸方向。

## 8. 实验方案与验证方法

### 8.1 基础功能验证

1. 内核能成功启动；
2. 串口打印正常；
3. 异常可捕获并打印现场；
4. 时钟中断周期稳定；
5. 内核线程切换正常。

### 8.2 用户态验证

1. 用户程序能被装载；
2. 用户态能通过系统调用输出字符；
3. 非法访问能正确触发异常；
4. 多个用户任务能被调度执行。

### 8.3 性能与轻量性验证

可选指标包括：

- 启动时间；
- 上下文切换开销；
- 中断响应时间；
- 内核镜像大小；
- 用户程序装载时间；
- 内存占用情况。

### 8.4 稳定性验证

- 长时间时钟中断稳定运行；
- 多任务反复切换；
- 系统调用压力测试；
- 异常恢复一致性测试。

## 11. 结论

“面向自研 LoongArch CPU 的应用承载型轻量内核设计与实现”是一个兼具研究价值、工程意义和竞赛可操作性的课题。LoongArch 已具备较明确的架构文档和主线生态参考，说明其在异常、中断、虚拟内存和启动等方面能够支撑系统软件开发。[1]  
相比直接移植完整 Linux，从零到一设计一个轻量但具备应用承载能力的内核，更适合自研 CPU 的早期 bring-up、教学与竞赛验证，也更利于理解软硬件协同设计的关键机制。

从实施路径上看，该方案最适合采用“分阶段、分层次、先闭环后增强”的思路：先实现最小内核闭环，再逐步扩展到用户态、系统调用、装载器和虚拟内存。这样既能降低工程风险，也更容易形成阶段性成果。因此，综合架构基础、工程路径与实现难度分析，本方案具有较强的可行性。
