# 调研报告
 **王允**
## 题目

面向移动端的轻量 AI 用户态调度引擎设计与实现：基于 Android 的 KV Cache 优化与 Remote Kernel 协同

## 摘要

本调研面向课程大作业中的系统创新实践，聚焦“移动端设备资源受限”与“大模型推理缓存开销大”两类核心矛盾。基于 AIOS v0.3.0 与 Cerebrum SDK 的 Remote Kernel 模式，提出一条降范围但可落地的实现路径：以 Android 为主平台实现轻量用户态调度引擎，保留 scheduler、memory manager、tool manager 三个核心模块，并将 KV Cache 作为主优化对象，构建“内存缓存 + 文件缓存 + 远端卸载”的分层管理机制。iOS 端仅保留概念验证，避免后台守护与沙箱复杂度。调研结论认为，该方向在技术、资源、时间和风险维度均具备较高可行性，能够在课程周期内交付可演示、可评测、可复现的系统成果。

## 关键词

移动端 AI 系统；用户态调度；KV Cache；分层缓存；Remote Kernel

## 一、研究背景与问题定义

随着大模型能力向端侧延伸，移动端 AI 应用面临两类瓶颈：

1. 设备内存与计算资源有限，难以支撑长上下文与多任务推理。
2. 推理过程中的 KV Cache 占用快速增长，导致频繁换入换出与延迟抖动。

AIOS v0.3.0 提供了 Remote Kernel 模式，允许资源受限设备仅运行轻量 SDK，将重计算卸载到远端节点。这为课程项目提供了明确机会：不做完整内核移植，而是在用户态做“调度与缓存管理”创新。

本调研将问题收敛为：

- 如何在 Android 端构建可运行的轻量调度引擎。
- 如何在有限内存下提升 KV Cache 命中率并控制延迟。
- 如何通过本地与远端协同，平衡响应速度与资源消耗。

## 二、调研目标与范围收敛

为确保课程周期内可交付，本方案采用降范围策略：

1. 平台范围：Android 主实现，iOS 仅概念演示。
2. 模块范围：仅实现 scheduler、memory manager、tool manager 三个核心模块。
3. 优化重点：优先完成 KV Cache 分层管理、动态逐出与低比特量化。

对应的预期交付包括：

- Android 端后台调度 Demo。
- 可复现实验脚本与性能数据。
- 调研报告与演示视频。

## 三、技术路线设计

### 3.1 系统总体方案

系统采用“本地轻量调度 + 远端重计算”混合模式：

1. 本地侧负责任务编排、工具调用、缓存管理和状态维护。
2. 当上下文或负载超过阈值时，将重计算请求卸载至 Remote Kernel。
3. 通过统一缓存元数据管理本地与远端状态切换。

### 3.2 KV Cache 优化路径

拟采用三层缓存结构：

1. 内存层：LRU 管理 + 低比特量化，保障热点数据低延迟访问。
2. 文件层：SQLite 或 mmap 文件缓存，承接冷数据。
3. 远端层：通过 Remote Kernel 兜底，避免本地内存失控。

策略上强调三点：

- 动态阈值逐出：根据设备可用内存动态调整驱逐门限。
- 成本感知迁移：综合命中概率与重计算代价决定迁移方向。
- 质量约束量化：在可接受精度损失范围内换取显著内存收益。

### 3.3 Android 实现路径

Android 端技术栈建议如下：

1. Foreground Service 负责后台持续调度。
2. Python SDK 嵌入用于快速联通 AIOS 能力。
3. Rust + NDK 承载 KV Cache 性能热点逻辑。
4. 系统 API 接入通知、文件、传感器等能力，形成可演示场景。

### 3.4 iOS 端策略

iOS 仅做 Remote Kernel 调用演示：

1. 使用轻量 Swift 界面发起远端推理请求。
2. 不实现后台守护，不引入复杂嵌入式运行时。
3. 目标是展示跨端一致的协同能力，而非完整端侧引擎。

## 四、可行性分析

### 4.1 技术可行性

1. 依赖成熟度较高：AIOS/Cerebrum、Android 工具链、Rust 移动端编译路径均有公开实践。
2. 架构复杂度可控：通过降范围将核心难点集中在缓存与调度，不扩散到系统底层移植。
3. 验证链路清晰：可在模拟器与本地远端节点上完成端到端测试。

### 4.2 资源可行性

1. 硬件要求低：普通开发机即可支撑模拟器与小模型验证。
2. 软件成本低：依赖以开源工具为主，额外经济投入可忽略。

## 五、风险识别与应对

### 5.1 跨语言与跨平台集成风险

风险：Rust、Python、Android 之间的类型转换与内存管理复杂。

应对：

1. 先打通最小调用链，再逐步扩展接口。
2. 为 FFI 边界增加统一数据结构与错误码封装。
3. 优先验证 Android 主链路，iOS 不强制共享底层实现。

### 5.2 缓存策略效果不稳定

风险：逐出阈值与量化粒度设置不当，可能导致命中率下降或精度波动。

应对：

1. 先实现规则型基线策略，再增加动态调节。
2. 建立固定评测集，持续跟踪延迟、内存和质量指标。

### 5.3 Android 后台保活与资源约束

风险：后台服务可能被系统策略限制，影响演示稳定性。

应对：

1. 演示场景优先保证链路闭环，不追求长期驻留。
2. 控制后台负载，降低异常资源占用概率。

### 5.4 远端协同稳定性

风险：本地与远端通信抖动影响响应一致性。

应对：

1. 固定远端版本与接口。
2. 增加重试与降级路径，确保可演示性优先。

## 六、相关研究启发

在“面向边缘设备多智能体推理的持久化 Q4 KV Cache”相关工作中，核心启发包括：

1. 通过低比特量化与持久化，可显著提升缓存容量并降低重复 prefill 开销。
2. 若能将缓存恢复延迟隐藏在多智能体交错执行过程中，可进一步提升端到端吞吐。
3. 缓存管理能力应抽象为模型无关的系统层能力，而非绑定单一模型实现。

这些结论与本项目的“分层缓存 + 远端协同”路线高度一致，可作为实验指标设计的重要依据。

综合评估认为，本选题具备较高可行性与课程创新价值：

1. 问题真实且聚焦，工程落地路径明确。
2. 降范围后难度可控，能够在课程周期内完成闭环。
3. 结果可通过延迟、内存、命中率等指标进行量化评估。

3. 最后补齐 iOS 概念演示与文档，完成最终交付闭环。
