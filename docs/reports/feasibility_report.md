# 面向 LLM 推理服务的 KVCache 分配、复用与驱逐协同优化可行性报告

## 摘要

随着大语言模型（Large Language Models, LLMs）在智能问答、代码生成、多轮对话、文档理解等任务中的广泛应用，面向在线服务场景的高效推理已成为当前研究和工程实践中的关键问题。与训练阶段相比，LLM 推理服务更强调低时延、高吞吐、显存利用率和多请求并发处理能力。在基于 Transformer 的自回归生成过程中，模型会为历史 token 保留 Key/Value 状态，以避免在后续解码时重复计算，这部分缓存通常被称为 KVCache。KVCache 的引入大幅降低了解码阶段的计算成本，但同时也带来了新的系统挑战：随着上下文长度增长和并发请求增多，KVCache 将持续占用大量显存资源，其分配策略、共享复用方式以及驱逐回收机制会直接影响推理系统的整体性能。

现有工作已经从多个角度证明了 KVCache 优化的重要性。例如，vLLM 通过 PagedAttention 将 KVCache 分块管理，从而降低显存碎片；Automatic Prefix Caching 通过哈希、引用计数和 LRU 驱逐机制实现共享前缀复用，减少重复 prefill 计算。然而，这些优化大多集中于某一局部环节，例如专注于分配器设计、前缀缓存命中或局部驱逐策略，而较少从服务系统整体角度考虑 KV block 的完整生命周期管理。在真实 LLM 推理服务中，KVCache 的分配、复用与驱逐并非彼此独立，而是存在明显的耦合关系：分配粒度影响共享机会，共享状态影响驱逐代价，驱逐策略又反过来影响后续复用效率。因此，仅进行单点优化难以充分挖掘 KVCache 的整体收益。

基于上述问题，本文提出一个面向 LLM 推理服务的 KVCache 生命周期管理思路，即将 KVCache 视作一个贯穿请求执行全过程的动态对象，统一考虑其分配、共享复用和驱逐回收过程。本项目拟在现有开源推理框架 vLLM 的基础上，对其 KV block 管理、前缀缓存和驱逐机制进行分析，并探索一种协同优化方案：在块式分配基础上，增强前缀复用能力，支持更复杂的“共享后逐步分叉”请求模式；在驱逐阶段，不再仅依据最近最少使用策略，而综合考虑复用潜力、共享状态、前缀位置和重算代价等因素，从而构建面向真实服务场景的 KVCache 协同优化框架。

综合研究价值、技术路径、工程条件和实验设计来看，本项目具有较强的可行性。后续工作将围绕 vLLM 源码分析、机制抽象、原型实现和实验验证四个方面展开，以期在现有 KVCache 优化体系上提出更系统、更具服务导向的协同优化方案。

**关键词：** 大语言模型推理服务；KVCache；前缀缓存；缓存驱逐；vLLM；协同优化；生命周期管理

## 1. 引言

### 1.1 研究背景

近年来，大语言模型的快速发展显著推动了自然语言处理、代码智能、多模态理解等领域的技术进步。特别是以 Transformer 为基础的生成式模型，在机器翻译、文本摘要、开放问答、对话系统以及代码补全等任务中取得了优异表现。随着模型规模不断扩大，LLM 已逐步从实验室研究走向实际部署，成为智能服务系统的重要组成部分。然而，在真实应用场景中，用户往往通过 API 或在线服务与模型交互，因此模型的“推理服务能力”已成为继训练能力之后的另一个核心问题。

与离线批量推理不同，在线 LLM 推理服务通常面临多方面压力：一方面，用户请求具有突发性和高并发特征，系统需要在有限硬件资源下尽可能提高吞吐率；另一方面，交互式服务对首 token 时延（Time To First Token, TTFT）和整体生成延迟高度敏感，任何不合理的资源调度都可能显著影响用户体验。同时，随着长上下文推理、多轮对话和检索增强生成等应用的发展，单个请求所需的上下文长度不断增加，导致推理阶段的显存开销快速上升。如何在保证服务质量的前提下高效利用显存，已经成为 LLM Serving 系统设计中的关键挑战。

在 Transformer 模型的自回归生成过程中，模型在处理当前 token 时，需要访问所有历史 token 的注意力状态。为了避免在每一步生成时重复计算之前所有 token 的中间结果，推理系统通常会将历史 token 对应的 Key 和 Value 张量保留下来，并在后续解码过程中直接复用，这部分缓存即为 KVCache。KVCache 是现代大模型推理服务得以高效运行的基础机制之一，它显著降低了解码阶段的计算量，使得模型可以在较低计算代价下进行逐 token 生成。然而，这种“以空间换时间”的机制也使得推理服务对显存资源的依赖进一步增强。

当上下文长度增加时，KVCache 的占用会近似随 token 数增长；当并发请求数增加时，不同请求持有的 KVCache 会进一步叠加；当服务中存在大量相似提示词或共享系统提示时，如果系统无法有效识别并复用这些前缀缓存，就会产生大量重复 prefill 计算和重复缓存分配。由此可见，KVCache 不仅是一个“模型中间状态缓存”问题，更是一个跨越请求调度、显存管理和服务优化的系统问题。

正因如此，近年来围绕 KVCache 的管理机制已经成为 LLM 推理系统优化的重要研究方向。工业界和学术界已经提出多种方法，例如通过块式或分页式结构降低内存碎片，通过共享前缀缓存减少重复计算，通过更灵活的调度提升多请求服务效率等。这些工作为提升推理系统性能提供了重要基础，也表明 KVCache 优化具有现实价值和广阔研究空间。

### 1.2 问题提出

虽然已有研究从不同角度优化了 KVCache，但现有方法仍存在明显局限。首先，许多工作采用的是“单点优化”思路，即分别针对某一个局部问题进行改进。例如，有的工作关注如何更高效地分配 KVCache 内存，有的工作关注如何识别相同前缀以实现复用，有的工作关注如何设计更合适的驱逐策略。然而，在实际推理服务中，这几个问题并不是相互独立的，而是彼此耦合、相互影响的。

KVCache 的**分配方式**决定了缓存的组织粒度和复用单位。如果缓存被切分为固定大小 block，那么共享与回收就往往以 block 为基本对象；而 block 的大小和组织方式会直接影响内存利用率、共享机会以及后续调度复杂度。KVCache 的**复用机制**决定了系统能否避免重复 prefill 计算，以及复用发生时如何维护 block 之间的引用关系。一旦某些 block 被多个请求共享，它们就具有了更高的“保留价值”，这又会进一步影响驱逐策略。KVCache 的**驱逐机制**则决定了在显存紧张时哪些 block 应优先被回收，而一旦驱逐了未来高概率复用的共享 block，系统就可能在后续请求中付出更高重算代价。因此，分配、复用和驱逐三者实际上构成了一个动态耦合系统。

其次，现有前缀复用机制大多偏向于“完全相同前缀”的共享方式。例如，对于相同的系统提示词或完全一致的 prompt 前缀，可以通过哈希等手段快速判断并复用已有 KV block。但在真实服务场景中，更常见的情况是“共享前缀后逐步分叉”：多个请求在前半部分拥有公共上下文，而在某个位置后开始出现局部差异。例如，不同用户都基于同一模板发起提问，只修改了个别槽位；或者一批请求共享检索结果的开头片段，但后续追加了不同指令。在这些场景中，如果系统只支持严格的完全前缀匹配，那么大量潜在可复用的 block 无法被利用，导致缓存收益下降。

再次，现有驱逐策略在很多情况下仍较依赖简单的最近最少使用（LRU）思想，即优先淘汰最近较少访问的缓存对象。然而，对于 KVCache 而言，一个 block 是否值得保留，并不能仅由“最近是否被访问”来决定。一个共享 block 可能在短时间内没有被再次命中，但它位于多个请求共用前缀的前段，未来一旦有相似请求到来，其重用价值就会很高；反之，一个最近刚被访问但只服务于单个长尾请求的 block，其长期保留价值未必更高。因此，KVCache 的驱逐问题需要综合考虑更多因素，例如共享状态、前缀位置、未来复用潜力以及被驱逐后的重算代价等。

基于以上观察，可以发现：**LLM 推理服务中的 KVCache 管理，本质上并不是若干独立优化技巧的简单叠加，而是一个涉及分配、复用、驱逐协同设计的系统性问题。**

### 1.3 研究意义

从工程应用角度看，研究 KVCache 协同优化具有直接意义。首先，显存是 LLM 推理服务中最稀缺、最昂贵的资源之一。对于给定硬件平台，如果能够更高效地管理 KVCache，就可能在不增加硬件投入的情况下支持更长上下文、更高并发或更低延迟。其次，在多轮对话、检索增强生成和模板化推理等场景中，请求之间往往存在显著的前缀共享特性，如果能够充分复用 KVCache，将有效减少重复 prefill 计算，提高整体服务效率。再次，对于在线服务而言，减少不必要的缓存驱逐和重复构建，也有助于提升系统稳定性和服务质量。

从研究角度看，KVCache 管理问题兼具系统性和算法性。一方面，它涉及内存组织、请求调度、元数据管理和并发服务机制，是典型的系统优化问题；另一方面，它又可以在共享判定、生命周期建模和驱逐策略设计等方面引入更细粒度的分析与策略设计，具有较强的研究拓展空间。因此，围绕 KVCache 的统一生命周期管理展开研究，不仅能够回应当前 LLM Serving 的现实需求，也具备形成新方法、新机制和新实验结论的潜力。

### 1.4 本项目的研究思路

基于上述背景和问题分析，本文拟提出一个面向 LLM 推理服务的 KVCache 生命周期管理思路。该思路的核心在于：**不再将前缀缓存、内存分配和缓存驱逐分别视为彼此孤立的问题，而是将 KVCache 视作一个贯穿推理服务全过程的生命周期对象，对其分配、共享、分叉、驱逐和回收进行统一管理。**

4. 通过可控负载实验评估命中率、重算开销、延迟和显存利用率，验证方案有效性与适用边界。

## 2. 研究背景与问题提出

### 2.1 LLM 推理服务的基本过程

大语言模型推理服务通常基于 Transformer 架构，采用自回归生成方式逐步输出文本。给定一个输入 prompt，模型首先对输入序列进行完整前向计算，得到各层注意力与前馈网络的中间结果，并形成对应的上下文表示；随后，模型再以逐 token 的方式进行解码，每生成一个新 token，就将其追加到历史序列中，并基于全部历史上下文继续预测下一个 token。这一过程一直持续到达到停止条件，如遇到结束符或达到最大生成长度。

在服务场景下，单次请求通常可分为两个阶段：**Prefill 阶段**和 **Decode 阶段**。Prefill 阶段指模型处理用户提供的已有上下文，将整段 prompt 一次性输入模型并计算所有层的 Key/Value 状态；Decode 阶段则是基于已有上下文状态逐步生成后续 token。对于长 prompt 而言，Prefill 阶段需要对所有输入 token 进行完整前向计算，其计算量较大；而 Decode 阶段虽然每一步只生成一个 token，但会频繁调用注意力机制，并持续依赖之前保存的历史状态。

在推理系统设计中，Prefill 和 Decode 的资源特点并不相同。Prefill 更偏向于大批量计算，对 GPU 算力和吞吐敏感；Decode 则更偏向于高频小步执行，对缓存状态管理和调度开销更加敏感。对于现代在线推理服务而言，如何平衡这两个阶段的资源消耗，并减少重复计算，是系统优化的核心问题之一。

如果进一步从服务系统角度比较二者，可以看到两阶段对资源与缓存管理提出了截然不同的要求。Prefill 的主要矛盾是“已有上下文很长、但许多前缀可能重复”；Decode 的主要矛盾则是“每一步计算量不大，但会持续占用 KVCache 并放大调度开销”。因此，任何只针对其中一个阶段的优化，都容易在另一个阶段暴露新瓶颈。

| 维度 | Prefill 阶段 | Decode 阶段 |
| --- | --- | --- |
| 计算模式 | 对整段输入做大批量前向计算 | 逐 token 小步生成 |
| 主要瓶颈 | 重复 prefill、首 token 延迟 | KVCache 驻留、调度频率、尾延迟 |
| 对缓存的需求 | 希望尽可能命中共享前缀 | 希望尽可能少驱逐、少重建 |
| 对系统的压力 | 算力、带宽、批处理组织 | 显存、状态维护、并发管理 |
| 优化重点 | 前缀复用、chunked prefill | 生命周期管理、驱逐与回收 |

这也是本文强调“统一生命周期管理”的直接原因：Prefill 和 Decode 虽然阶段不同，但它们共享同一份 KV block 资产，不能割裂优化。

### 2.2 KVCache 的作用及其重要性

在 Transformer 的自注意力计算中，每个 token 都需要与之前所有 token 的 Key/Value 进行交互。如果在每一步生成时都重新计算整个历史序列的注意力状态，那么代价将非常高。为了解决这一问题，推理系统通常在每一层保存历史 token 对应的 Key 和 Value 张量，在后续生成时直接读取这些缓存，而不再重复计算先前 token 的中间结果。这种缓存机制就是 KVCache。

KVCache 的引入极大地提升了自回归生成效率。特别是在 Decode 阶段，新 token 的生成只需计算当前 token 对应的 Query，并与缓存中的历史 Key/Value 进行注意力运算，而不再对整个历史序列做完整前向传播。因此，KVCache 已经成为几乎所有高性能 LLM 推理框架的标准配置。

然而，KVCache 的代价也十分明显。其显存占用会随以下因素增加：

1. 模型层数；
2. 注意力头数与 hidden size；
3. 上下文长度；
4. 并发请求数；
5. 生成长度。

当模型规模较大、上下文较长或并发请求增多时，KVCache 占用的显存往往会超过模型参数本身的占比。对于面向服务部署的系统来说，这意味着即便模型参数已经被成功装载到 GPU 中，系统实际可支持的请求规模仍然会受到 KVCache 的强烈约束。因此，KVCache 管理效率直接决定了服务系统的可扩展性。

如果用更定量的方式描述，对于 decoder-only 模型，单个请求在长度为 $T$ 的上下文上所需的 KVCache 近似满足：

$$
M_{\text{KV,req}} \approx 2 \times L \times H_{kv} \times D_{head} \times T \times s,
$$

其中，$L$ 为层数，$H_{kv}$ 为 KV 头数，$D_{head}$ 为每个头的维度，$s$ 为单元素字节数，例如 FP16/BF16 常取 2。若同时存在 $R$ 个并发请求，则总 KVCache 开销可近似写为：

$$
M_{\text{KV,total}} \approx \sum_{r=1}^{R} 2 \times L \times H_{kv} \times D_{head} \times T_r \times s.
$$

这个近似式已经足以说明三个关键事实：第一，KVCache 对上下文长度是近似线性增长的；第二，并发请求之间的缓存开销会直接叠加；第三，哪怕模型参数已经成功加载，真正限制并发能力的往往仍然是剩余可用于 KVCache 的显存空间。

进一步地，对于存在重复前缀的请求集合，如果不做复用，系统需要为每个请求单独承担前缀 prefill 代价；若设公共前缀长度为 $P$，分叉后新增长度为 $\Delta_i$，则重复计算代价可粗略理解为从

$$
\sum_{i=1}^{R} (P + \Delta_i)
$$

下降为

$$
P + \sum_{i=1}^{R} \Delta_i.
$$

因此，前缀缓存的价值不仅体现在“省下一些显存”，更体现在避免重复构建本可共享的历史状态。这个收益会在模板化请求、共享系统提示和多轮检索增强场景中进一步放大。

### 2.3 KVCache 管理中的主要挑战

虽然 KVCache 能显著减少重复计算，但在真实服务系统中仍面临四类核心挑战：

1. **显存预算紧张。** 模型参数、运行时激活和 KVCache 共享同一显存池；保留过多会压缩并发空间，驱逐过快又会引发重算。
2. **分配与碎片化矛盾。** 连续分配容易产生外部碎片，块式/分页虽可缓解碎片，但会增加映射与元数据管理开销。
3. **共享与一致性开销。** 前缀复用可降低 prefill 成本，但需要维护引用关系、分叉路径与共享对象的一致性语义。
4. **驱逐决策困难。** 仅按近期访问做淘汰容易误删高价值共享块，导致后续命中下降与重算上升。

同时，KVCache 管理与请求调度强耦合：continuous batching、chunked prefill、preemption 等机制会持续改变 block 的驻留时长与增长速度。因此，单独优化某个局部结构，往往难以稳定改善系统指标。

### 2.4 分配、复用与驱逐的耦合性

在许多传统缓存系统中，分配、命中和淘汰往往可以相对独立地分析。但在 LLM 推理服务中，KVCache 的分配、复用和驱逐之间存在更紧密的耦合关系。

从分配角度看，系统如何组织 KVCache，将直接影响复用粒度。如果一个请求的缓存以整段连续内存形式存在，那么共享和回收通常只能粗粒度进行；如果使用固定大小 block，则既方便部分共享，也便于按块回收，但同时需要维护额外映射关系。换言之，**分配方式决定了复用和驱逐的操作对象**。

从复用角度看，一旦多个请求共享某些前缀 block，这些 block 的“价值”就不再只取决于当前是否被访问，而是取决于其引用关系、共享深度以及未来再次命中的可能性。共享越广泛、位于前缀越靠前的 block，通常越值得保留。因此，**复用状态会反过来影响驱逐优先级**。

从驱逐角度看，当系统回收某些 block 后，未来一旦有新请求需要相同前缀，就必须重新进行 prefill 计算并重新分配 block。这意味着驱逐行为不仅释放显存，也可能损害后续复用机会和服务效率。因此，**驱逐策略会进一步改变后续分配压力和复用收益**。

由此可见，分配、复用与驱逐形成了一个闭环：分配方式影响共享能力，共享状态影响驱逐价值，驱逐决策又影响未来的分配压力和复用收益。这一耦合关系正是本项目试图统一建模和协同优化的核心出发点。

### 2.5 研究问题的提出

基于上述背景，本文提出如下核心问题：

**在面向 LLM 推理服务的场景中，能否将 KVCache 视作一个具有完整生命周期的对象，对其分配、复用和驱逐进行统一建模与协同优化，从而提升缓存利用效率、降低重算成本并改善系统整体服务性能？**

围绕这一问题，本文进一步关注以下几个子问题：

1. 如何描述 KV block 从创建到回收的生命周期过程？
2. 如何在现有 block/page 机制基础上增强前缀共享能力，尤其是支持共享后逐步分叉的请求模式？
3. 如何设计一种不仅依赖 LRU，而能综合考虑共享状态、复用潜力和重算代价的驱逐策略？
4. 如何在现有开源推理框架（如 vLLM）上验证上述机制的可实现性与有效性？

这些问题共同构成了本项目的研究基础，也为后续技术方案和实验设计提供了明确方向。

## 3. 相关工作

### 3.1 基于 Transformer 的 LLM 推理缓存机制

Transformer 架构已成为当前大语言模型的主流基础结构。其核心注意力机制要求每个位置在计算时访问历史序列的表示，因此在自回归生成场景中，如果每一步都重新计算全部历史 token 的中间表示，推理开销会随着序列长度快速增长。KVCache 的引入使得模型可以在首次处理历史 token 后，将各层的 Key 和 Value 张量保存下来，并在后续生成阶段直接复用，从而避免重复计算。可以认为，没有 KVCache，高性能的在线 LLM 推理服务几乎无法成立。

不过，KVCache 虽然降低了计算代价，却将系统瓶颈从“算力不足”部分转移到了“显存管理复杂”和“缓存利用效率不足”上。特别是在长上下文、多轮对话和高并发服务场景中，KVCache 的规模巨大，且生命周期复杂。因此，围绕 KVCache 的组织、管理和优化逐渐成为 LLM Serving 研究的一个重要方向。

### 3.2 PagedAttention 与块式 KVCache 管理

为了解决传统连续内存分配方式带来的显存碎片问题，vLLM 提出了 PagedAttention 思路，将每个请求的 KVCache 切分为固定大小的 block，并采用类似虚拟内存分页的思想进行管理。该方法的关键不在于改变注意力计算本身的数学形式，而在于改变 KVCache 的组织方式：通过将逻辑上的连续序列映射到物理上的离散 block，可以避免为每个请求分配过大的连续显存区间，从而提升显存利用率并减少外部碎片。

PagedAttention 的提出具有重要意义。首先，它使 KVCache 从“整段连续对象”转变为“可管理的块对象”，为后续更灵活的复用、共享与回收提供了结构基础。其次，block 化管理更适合多请求并发场景，因为不同请求可以按需增长其缓存，而不是预留大量无效空间。再次，块式组织方式为前缀复用创造了条件：如果多个请求共享同一前缀，那么它们理论上可以直接共享对应的某些 block，而不必复制整段缓存。

然而，PagedAttention 并不是没有代价。它引入了更复杂的 block mapping、索引管理和 kernel 层适配问题，也意味着系统需要维护从逻辑 token 序列到物理 block 布局的额外元数据。这说明，即使块式分配减轻了碎片问题，KVCache 管理的系统复杂度仍然显著存在。

### 3.3 Automatic Prefix Caching 与前缀复用

在许多真实推理场景中，请求之间存在较强的前缀重叠。例如，同一个系统提示、同一段文档开头、同一批模板请求中的固定前缀等。若系统能识别这些相同前缀，并复用已计算好的 KVCache，就可以避免重复的 prefill 计算，从而降低时延并提升吞吐。

vLLM 在此基础上引入了 Automatic Prefix Caching 机制。其基本思路是：对 KV block 进行哈希标识，并通过引用计数维护共享状态；当新请求到来时，系统可根据已有 block 的哈希信息判断当前请求是否存在可复用前缀；若存在，则直接复用对应 block，而不必重新计算这些前缀内容。与此同时，系统还会结合 LRU 等缓存驱逐策略，对长期未使用的 block 进行回收。

Automatic Prefix Caching 证明了前缀复用在工程实践中的价值。相比完全不利用前缀重叠的系统，它能够有效减少重复计算，尤其适用于具有高模板相似性或共享系统提示的场景。但其局限也比较明显：现有机制更适合处理“完全相同的前缀”，而对“局部分叉”“部分共享”“chunk 级重叠”等更复杂模式支持有限。一旦前缀中某个位置发生变化，后续缓存很可能无法再直接复用，从而降低缓存收益。

### 3.4 动态内存管理与 vAttention 的启发

除了 vLLM 之外，研究者也开始从更底层的动态内存管理视角审视 KVCache 问题。相关工作指出，PagedAttention 虽然在逻辑上缓解了碎片和空间浪费，但也确实提高了推理系统和注意力计算内核的实现复杂度。这表明，块式管理并不是“零成本收益”，而是一种系统层面的取舍：它以额外的管理逻辑和实现复杂度，换取更好的显存利用率和更高的并发服务能力。

这类工作为本项目提供了重要启发：KVCache 管理问题不能只看“分配是否更省空间”，还必须从系统整体视角考虑实现代价、调度影响和复用收益。换言之，单独讨论分配器、前缀缓存或驱逐策略都不够，因为这些机制共同作用于一个动态的推理服务系统。

### 3.5 现有工作的局限性

综合来看，现有相关工作已经为 KVCache 优化打下了坚实基础，但仍存在以下不足：

1. **单点优化较多，统一视角不足。**  
   许多方法分别从分配、复用或驱逐某一侧切入，但较少将三者作为一个耦合系统统一分析。

2. **共享机制偏向完全前缀匹配。**  
   对于“共享后逐步分叉”的真实请求模式，现有方法的利用率仍有限。

3. **驱逐策略通常较为简单。**  
   即使考虑了 LRU 或引用计数，也较少同时纳入复用潜力、重算代价、共享深度和前缀层次信息。

4. **对 KV block 生命周期缺少系统抽象。**  
   当前许多系统把 block 当成“被分配出来后等待回收的静态对象”，而不是具有创建、共享、分叉、冷却和淘汰等状态变化的生命周期对象。

本项目正是在这些局限之上提出研究设想，即试图构建一个统一的 KVCache 生命周期管理框架，以协同方式处理分配、复用与驱逐问题。

### 3.6 本项目与现有工作的关系

现有研究可概括为三类：更省空间的放置、更多前缀命中、以及显存压力下的淘汰/压缩。它们分别覆盖分配、复用与回收，但较少从统一生命周期角度协同设计。

| 研究方向 | 典型思路 | 已有优势 | 尚存空白 |
| --- | --- | --- | --- |
| 块式/分页式管理 | 将 KVCache 切为 block 或 page | 降低碎片、按需增长、便于部分回收 | 很少直接表达 block 的长期复用价值 |
| 前缀复用 | 基于哈希或前缀匹配复用已有 block | 减少重复 prefill，改善 TTFT | 对局部分叉和分层共享支持有限 |
| 压缩/驱逐 | 压缩部分 KV 或设计淘汰策略 | 缓解显存压力，扩大可服务规模 | 往往缺少与分配、共享状态的协同建模 |

本项目并非替代上述路线，而是将其串联：以块式管理作为结构基础，以前缀复用扩大共享机会，再用生命周期感知驱逐决定长期保留对象。关注点从“单点是否有效”转向“组合后是否稳定增益”。

## 4. 项目研究问题定义

### 4.1 研究对象界定

本项目的研究对象是**面向 LLM 推理服务场景的 KVCache 管理机制**，具体聚焦于 KVCache 在服务运行过程中的三个关键环节：

1. **分配（Allocation）**：系统如何为请求按需分配 KV block；
2. **复用（Reuse/Sharing）**：系统如何在请求之间共享已有前缀缓存；
3. **驱逐（Eviction/Reclamation）**：系统如何在显存压力下回收部分缓存。

与仅分析某一环节不同，本项目关注的是三者之间的耦合关系，以及如何在统一框架下实现协同优化。

### 4.2 服务场景下的典型请求模式

为了更准确地刻画问题，本项目考虑以下几类典型请求模式：

#### 4.2.1 完全共享前缀模式

多个请求具有完全相同的前缀，例如相同系统提示、相同模板开头等。该场景是现有前缀缓存机制最容易利用的情况，也是现有系统的主要受益场景。

#### 4.2.2 局部分叉模式

多个请求在前半部分共享相同上下文，但在某一位置后开始出现差异。例如相同任务模板下，个别字段不同。这类场景在实际服务中非常常见，但现有系统对其利用往往不充分。

#### 4.2.3 逐步分叉模式

一批请求并不是在同一位置一次性分叉，而是在多个阶段逐步出现差异。此时共享关系不是简单的“共享/不共享”二值状态，而更接近树状或分层结构。

#### 4.2.4 Chunk 级部分重叠模式

不同请求在 token 级别未必构成严格前缀关系，但在某些 chunk 级别存在相似或相同片段。对于这类更细粒度的共享机会，传统的严格前缀复用机制往往难以直接利用。

#### 4.2.5 混合冷热负载模式

真实线上服务往往不是单一负载，而是“热门模板请求 + 少量长尾请求 + 不同到达时间”同时存在。此时，一部分公共前缀会在短时间内被频繁命中，形成明显的热点 block；另一部分请求则只出现一次，却可能占用很长上下文。混合冷热负载是最能检验驱逐策略有效性的场景：如果系统只偏向热点，长尾请求可能频繁抖动；如果系统只追求公平保留，热门共享 block 又可能过早被淘汰。

### 4.3 KVCache 生命周期对象的定义

本项目提出将 KVCache 视为一个**具有完整生命周期的对象**。在这一视角下，一个 KV block 不是静态内存片段，而是会经历多个状态阶段的动态实体。典型状态可包括：

- **创建（Create）**：首次为请求分配 block；
- **填充（Populate）**：block 中写入某段前缀或生成 token 对应的 KV 数据；
- **共享（Share）**：block 被多个请求共同引用；
- **分叉（Fork）**：在共享前缀基础上，某个请求产生新分支；
- **写时复制（Copy-on-Write）**：共享状态下出现修改需求时进行增量复制；
- **冷却（Cooling）**：短期内不活跃，但仍可能被未来请求复用；
- **驱逐（Evict）**：在显存压力下被回收；
- **重建（Recompute/Rebuild）**：被驱逐后重新通过 prefill 计算恢复。

通过引入生命周期对象视角，可以更自然地统一描述 block 的价值变化过程。

在上述定义基础上，生命周期模型还需要满足若干实现约束。首先，只有已经填满、且内容稳定的 block 才适合作为可哈希、可共享的缓存对象；未填满的尾块通常仍处于“构建中”状态，不宜直接进入共享池。其次，共享 block 一旦被多个请求引用，就应被视为逻辑上的只读对象，后续分叉应通过追加新 block 或写时复制完成，而不是原地修改。再次，只有引用计数为 0 的 block 才应进入真正可驱逐集合；否则驱逐动作将破坏正在运行请求的正确性。最后，block 被驱逐后，其哈希、热度和分叉关系等元数据也应同步失效，否则会带来错误命中或脏统计问题。

这些约束的意义在于，它们把“生命周期建模”从抽象概念落实为可以映射到工程实现的状态机。只有在状态可判定、转移可约束的前提下，后续的共享管理和驱逐决策才具备可操作性。

### 4.4 本项目拟解决的核心问题

基于上述定义，本项目拟重点解决以下问题：

1. 如何建立 KV block 的生命周期状态模型及状态转移规则？
2. 如何支持比严格前缀复用更灵活的共享机制？
3. 如何定义 block 的“长期价值”或“生命周期价值”，使其能指导驱逐决策？
4. 如何将分配、共享和驱逐策略统一到一个可实现的管理框架中？
5. 如何在 vLLM 上进行最小可行原型验证？

为了使问题边界足够清晰，本文还做出以下约束：研究对象以 decoder-only 文本生成模型为主；原型验证优先面向单机单 GPU 场景，而不是一开始就覆盖分布式并行；初始阶段不改动 attention 数学语义和 CUDA kernel 的核心实现，而是优先在 block 元数据、复用判定和驱逐策略层面做增量改造；对于不构成严格前缀关系的 chunk 级重叠模式，先作为扩展方向讨论，不作为最小可行原型的强制目标。这样可以显著降低工程复杂度，使项目更容易在课程周期内交付可验证结果。

## 5. 项目核心思想

### 5.1 从单点优化转向统一生命周期管理

现有优化往往分别处理碎片、命中和淘汰，但在线服务中的 block 价值是随时间变化的。本文因此采用统一视角：**把 KVCache 当作生命周期对象，协同优化分配、复用与驱逐。**

在该视角下，决策不只看“现在是否被访问”，还要看共享范围、前缀位置、未来复用概率及驱逐后的重算代价。

### 5.2 协同优化的基本逻辑

统一管理的逻辑可概括为三点：

1. 分配阶段优先选择有利于后续共享和分叉管理的粒度与组织方式；
2. 共享状态反向提供“保留价值”信号，影响驱逐优先级；
3. 驱逐决策会改变后续重建成本与分配压力，必须纳入长期收益评估。

### 5.3 共享后分叉的管理思路

在真实服务中，请求常呈现“共享前缀后分叉”。针对该模式，本文采用增量式管理思路：

- 允许多个请求共享已有 block；
- 当后续路径分叉时，仅为新增部分分配新 block；
- 必要时采用写时复制策略，减少无谓复制；
- 维护共享关系图或分层引用信息，为后续驱逐和统计提供依据。

其核心意义是把复用从“整段命中/不命中”扩展为“分层共享+增量生长”。

### 5.4 从“最近使用”到“生命周期价值”

传统 LRU 只反映时间局部性，难以覆盖 KVCache 的长期价值。本文引入“生命周期价值”概念，综合考虑：

- 最近使用情况；
- 共享引用数量；
- 未来复用概率；
- 前缀层级位置；
- 被驱逐后的重算代价；
- 所属分叉结构的重要性。

驱逐决策由此从单因素时间规则，扩展为面向服务目标的多因素评分。

### 5.5 协同优化的目标函数

从系统目标看，本文追求的并不是单一指标的极致最优，而是在延迟、吞吐、显存占用和重复计算之间取得更合理的平衡。可以将其抽象为一个多目标优化问题：

$$
J = \alpha \cdot L + \beta \cdot C + \gamma \cdot M - \delta \cdot U,
$$

其中，$L$ 表示请求延迟，$C$ 表示因驱逐或未命中而产生的重算代价，$M$ 表示显存压力，$U$ 表示由共享复用带来的长期收益。不同部署场景可以对应不同权重设置：交互式助手更重视 TTFT 和尾延迟，高并发离线服务更重视吞吐和显存效率，而研究性原型则更关注机制是否稳定可解释。

因此，本文提出的生命周期管理框架本质上是一种“服务目标感知”的缓存治理方式。它既不把前缀命中率当作唯一目标，也不把显存占用最小化视为唯一正确答案，而是试图让 block 的保留、共享和回收更贴合真实服务收益。

## 6. 项目创新点

### 6.1 统一生命周期视角

从统一生命周期角度建模 KVCache，将 block 的分配、共享、分叉与回收纳入同一框架，避免仅做局部改良。

### 6.2 分配、复用与驱逐协同优化

强调三者耦合关系：分配粒度影响共享机会，共享状态影响驱逐价值，驱逐结果反过来影响重建成本与后续分配压力。

### 6.3 面向局部分叉的共享增强

面向“共享后分叉”场景扩展复用边界，支持增量分叉与写时复制，并将 chunk 级共享作为后续扩展方向。

### 6.4 生命周期感知的驱逐策略

驱逐策略不再仅依赖 LRU，而是联合考虑复用潜力、共享状态、前缀位置和重算代价，提升长期收益。

### 6.5 面向开源推理系统的原型验证路径

以 vLLM 为原型平台开展增量实现与验证，保证方案可评测、可复现、可工程落地。

## 8. 关键技术方案

### 8.1 生命周期建模方案

本项目拟将每个 KV block 视作一个具有多状态属性的实体，并为其维护如下信息：

- block 标识；
- 所属前缀或 chunk 标识；
- 当前引用计数；
- 最近访问时间；
- 历史复用次数；
- 所在前缀层级位置；
- 所属分叉路径信息；
- 重算代价估计值。

这些元数据将用于支撑共享管理和驱逐决策。

为了让这些信息真正服务于在线决策，元数据的设计必须满足“低开销、可增量更新、与 block 边界对齐”三个原则。可维护的信息及其作用如下表所示：

| 元数据 | 含义 | 主要更新时机 | 作用 |
| --- | --- | --- | --- |
| `ref_cnt` | 当前引用该 block 的请求数量 | 共享、释放、回收时 | 判定是否可驱逐 |
| `last_access_ts` | 最近一次命中或被请求引用的时间 | cache hit、append、复用时 | 反映时间局部性 |
| `hit_count` | 历史累计复用次数 | 前缀命中时 | 估计长期热度 |
| `share_degree` | 当前共享深度或共享请求数 | 新请求复用、请求结束时 | 区分独占块与热点共享块 |
| `prefix_depth` | block 位于共享前缀中的相对位置 | block 首次完成时 | 早期前缀通常更值得保留 |
| `recompute_cost` | block 被驱逐后重建所需代价估计 | prefill 完成或驱逐后 | 指导协同驱逐 |
| `branch_factor` | 其后续是否出现多分叉 | 分叉形成或合并统计时 | 识别“共享后分叉”结构 |

其中，部分字段可以直接复用现有 block 元数据或请求信息，部分字段可通过 side table 方式额外维护。这样做可以避免在第一阶段大幅修改 vLLM 的核心对象结构，降低侵入性。

### 8.2 共享复用增强方案

在现有严格前缀复用机制基础上，可考虑以下增强思路：

1. **Chunk 级共享**  
   不将整段前缀视作唯一复用单位，而允许按更小粒度 block/chunk 判断共享机会。

2. **共享后写时复制**  
   当共享 block 被多个请求引用后，若某一请求在后续发展中需要独立修改，仅对新增部分或冲突部分进行增量复制。

3. **增量分叉管理**  
   对共享前缀之后的不同路径进行结构化管理，使多个请求之间形成树状共享关系，而不是简单的线性独占关系。

考虑到项目周期和实现风险，本文更倾向于采用“分层增强”而非一次性实现所有高级功能：

1. **L0：严格前缀命中层。**  
   直接沿用现有 block hash 命中逻辑，作为稳定基线。

2. **L1：共享后分叉层。**  
   当多个请求在完整共享若干 block 后分叉，仅为新增尾部 block 分配新空间，并显式记录共享链与分支链的关系。

3. **L2：可选的 chunk 级近似层。**  
   对不构成严格前缀、但存在局部重叠的请求进行更细粒度探索。该层更适合作为后续扩展，而不应成为首版原型的阻塞项。

这种分层方式的优点是：即使 L2 暂时无法完成，L0 + L1 也已经足以验证“统一生命周期管理”在真实服务场景中的核心收益，尤其适合模板化 prompt 和共享系统提示的主流负载。

### 8.3 协同驱逐策略方案

驱逐策略可设计为一个综合评分函数，而非简单 LRU。示意性地，可将 block 的生命周期价值（保留价值）建模为：

- 最近访问热度；
- 历史复用频率；
- 当前共享度；
- 前缀位置权重；
- 重算代价；
- 分叉结构重要性。

根据综合评分确定驱逐优先级，使生命周期价值更高的 block 更倾向于保留。

在此基础上，可将 block 的“生命周期价值”定义为一个轻量级综合分数：

$$
KeepScore(b) =
w_1 \cdot \text{Heat}(b) +
w_2 \cdot \text{Share}(b) +
w_3 \cdot \text{Reuse}(b) +
w_4 \cdot \text{PrefixPos}(b) +
w_5 \cdot \text{Recompute}(b).
$$

其中，各项可归一化到 $[0,1]$ 范围：`Heat` 反映最近访问热度，`Share` 反映当前共享程度，`Reuse` 反映历史复用频率，`PrefixPos` 体现其在共享前缀中越靠前越重要，`Recompute` 反映驱逐后需要重新 prefill 的代价。驱逐时优先淘汰 `KeepScore` 更低的 block，而不是简单淘汰“最久未访问”的 block。

为了控制运行时开销，本项目不打算在每一次 token 访问时都重排全局候选集，而是采用“两阶段决策”：第一阶段保留现有 LRU 队列作为粗筛，快速给出一小批最可能被淘汰的候选；第二阶段仅在显存水位接近阈值时，对这批候选块计算生命周期价值分数，再决定最终驱逐对象。这种设计兼顾了工程现实与策略表达能力：既不完全推翻现有 vLLM 的块回收路径，也为协同驱逐提供了足够的策略空间。

### 8.4 最小可行原型实现思路

考虑到项目周期和实现复杂度，本项目可采用“最小可行原型”策略，即：

- 首先不重写整个分配器；
- 先在现有 vLLM block 管理逻辑上增加统计与元数据维护；
- 再尝试替换或扩展现有驱逐优先级；
- 在可行情况下进一步引入局部分叉共享逻辑。

这种方式可以降低工程风险，提高项目落地概率。

更具体地说，最小可行原型可以分三步推进：

1. **观测层。**  
   先为 block 增加或外挂生命周期统计信息，记录命中、共享、驱逐、重建等事件，确保后续策略有数据支撑。

2. **策略层。**  
   在不改动 attention kernel 的前提下，替换或扩展 block 的驱逐优先级计算方式，先验证“生命周期感知驱逐”是否有效。

3. **增强共享层。**  
   在前两步稳定后，再尝试将共享链与分叉链显式建模，逐步支持“共享后分叉”的请求模式。

这样的推进顺序有两个好处：一是每一步都有可独立验证的中间成果；二是即使后续共享增强部分未完全落地，前两步也足以支撑一份完整、可评测的课程项目成果。

### 8.5 实现中的正确性与性能约束

为了保证原型可落地，本项目在实现时需要坚持以下原则：

1. 不破坏现有 block table 的稳定性，避免因物理 block 重排影响已生成序列的索引关系；
2. 不在热路径中引入过重的 Python 对象分配和全局排序操作；
3. 共享 block 默认视为只读，任何分叉写入都通过追加或写时复制完成；
4. 优先在 Python 层的 manager、pool、scheduler 交界处改造，尽量不触碰 CUDA kernel；
5. 所有新增统计都应能够在关闭时退化为接近原始 vLLM 的行为，便于做 A/B 对比。

这些约束既是工程实施时的安全边界，也体现了本文“先做机制验证，再考虑大规模重构”的基本态度。

## 9. 基于 vLLM 的实现基础分析

### 9.1 vLLM 的适配性

vLLM 是一个面向高性能推理服务设计的开源框架，已具备较完善的 KVCache 管理机制。相较于很多仅支持单机离线推理的实现，vLLM 更强调在线服务场景中的吞吐与显存效率，因此与本项目问题高度契合。

### 9.2 vLLM 已有机制对本项目的支撑

目前，vLLM 已具备如下基础能力：

- 基于 `v1/core/kv_cache_utils.py` 的 `KVCacheBlock` 元数据定义，block 本身已包含 `block_id`、`ref_cnt`、`block_hash` 等核心字段；
- 基于 `v1/core/block_pool.py` 的 `BlockPool` 与 `FreeKVCacheBlockQueue`，负责 free block 队列、缓存块映射与 LRU 式回收次序；
- 基于 `v1/core/kv_cache_manager.py` 的 `KVCacheManager`，对上向 scheduler 暴露 `get_computed_blocks`、`can_fit_full_sequence`、`allocate_slots` 等接口；
- 基于 `v1/core/single_type_kv_cache_manager.py` 与相关 coordinator 的分组管理逻辑，支持不同 KV cache group 的统一协调；
- 基于 `v1/core/sched/scheduler.py` 的请求调度路径，可观察 prefix hit 对 `num_computed_tokens` 和调度行为的影响；
- 现有 prefix caching 统计与日志接口，可作为后续实验采集的基础。

尤其值得注意的是，当前 `BlockPool` 已经维护了“block hash -> cached block”的查找结构，并显式把 free block 队列按 LRU/驱逐顺序组织起来；`KVCacheBlock` 则已经提供引用计数和哈希失效的基本语义。这意味着本文所需的生命周期信息并非从零构造，而是可以在现有抽象上继续补充。

这些能力意味着，本项目不需要从零开始设计推理引擎，而可以在现有框架基础上研究其不足并尝试改进。

### 9.3 可切入的源码模块

从工程实现角度，本项目后续重点关注如下模块：

- `LLM` 与 `LLMEngine` 请求入口，用于梳理请求从 API 到 scheduler 的进入路径；
- `v1/core/sched/scheduler.py`，用于分析 prefix hit 后 `num_computed_tokens` 变化如何影响调度；
- `v1/core/kv_cache_manager.py`，作为 computed blocks 查询、容量判定与分配接口的主入口；
- `v1/core/block_pool.py`，负责 free block 队列、缓存映射和回收候选，是驱逐策略最直接的切入点；
- `v1/core/kv_cache_utils.py`，定义 `KVCacheBlock`、free queue 等基础元数据结构；
- `v1/core/single_type_kv_cache_manager.py` 及 coordinator 相关模块，用于理解多 group 情况下的块分配；
- `v1/attention/ops/prefix_prefill.py` 与相关 attention 路径，用于确认 prefix hit 后的计算行为没有被破坏。

结合当前源码还能得到两个对原型设计很重要的约束。第一，现有实现为了保持 block table append-only，会尽量避免在缓存命中后对已分配 block 的身份做激进重排，这提示我们应优先在元数据层建模，而不是贸然做物理级去重。第二，block hash 只对“已填满且稳定”的 block 有意义，说明共享与驱逐最好都以 full block 为基本单位，而不是以任意长度的尾块为单位。

通过聚焦这些核心模块，可以逐步完成从机制分析到原型改造的过渡。

### 9.4 建议的最小切入顺序

结合当前 vLLM v1 结构，一个较稳妥的改造顺序是：

1. 在 `KVCacheManager` 和 `BlockPool` 交界处补充统计与日志，先观测命中、驱逐和重建行为；
2. 通过 side table 维护 block 的生命周期价值字段，避免第一阶段直接改动过多核心类定义；
3. 在 `BlockPool` 的回收路径中引入低水位触发的二阶段驱逐决策；
4. 待上述机制稳定后，再向 scheduler 侧暴露更多共享/分叉信息，用于更精细的实验分析。

这种切入顺序可以最大限度复用现有 vLLM 抽象，同时把风险集中在 Python 层，减少对 kernel 和底层执行路径的影响。

因此，当前环境虽不面向大规模工业评测，但足以支撑“原型实现 + 机制对比 + 指标分析”的课程项目目标。

### 10.3 资源预算与实验边界

结合当前 8GB 显存条件，本项目更适合采用“分层验证”策略：用 `Qwen/Qwen3.5-2B` 完成 smoke test、仪表化和频繁 A/B 调试，用更大显存机器补充 8B 级对照；`max_model_len` 则优先设置在 1024 到 2048 的可控范围内，以便放大 KVCache 管理策略的差异而不至于频繁 OOM。对于特别强调长上下文的实验，可通过提高共享前缀比例、增加并发请求数而不是单纯拉长上下文长度，来更稳定地观察复用和驱逐行为。

在此基础上，后续验证阶段可将关键对比组迁移到 RTX 3090 服务器复跑，重点观察更高并发和更长上下文下的指标一致性，以确认协同策略收益不依赖于单一本地硬件条件。

因此，当前硬件虽然不适合做超大模型、超长上下文的工业级评测，但非常适合做“有控制变量的系统机制验证”。这与本文的研究目标是匹配的。

## 11. 实验设计与评估指标

### 11.1 实验目标

本项目实验的目标不是追求超大规模绝对性能，而是验证所提出的协同优化思路在典型服务场景中的可行性与潜在收益。具体关注以下问题：

- 是否能提高前缀复用率？
- 是否能减少无效驱逐和重算？
- 是否能在相同显存条件下支持更高缓存利用率？
- 是否能改善请求延迟或吞吐率？

### 11.2 请求工作负载设计

为体现不同机制的效果，实验拟构造以下几类请求集：

1. **完全共享前缀集**  
   多个请求共享相同系统提示或固定模板。

2. **局部分叉请求集**  
   多个请求在大段前缀相同后，从某个位置开始出现差异。

3. **逐步分叉请求集**  
   请求在多个位置逐渐分化，形成层次化共享结构。

4. **Chunk 级部分重叠请求集**  
   请求在某些局部片段相同，但不构成严格前缀关系。

为了让不同策略之间的差异更容易显现，工作负载还需要显式控制以下变量：共享前缀长度、分叉深度、请求到达模式、并发度以及生成长度。可采用如下实验矩阵：

| 负载类型 | 共享前缀长度 | 分叉方式 | 并发度 | 主要观察点 |
| --- | --- | --- | --- | --- |
| 完全共享 | 高 | 无分叉 | 4-16 | prefix hit 与 TTFT 改善 |
| 单点分叉 | 中高 | 在固定位置一次分叉 | 4-16 | 共享链保留价值、尾块分配效率 |
| 逐步分叉 | 中 | 多阶段分叉 | 4-12 | 分叉结构对驱逐的影响 |
| 混合冷热 | 不固定 | 热点模板 + 长尾请求 | 8-16 | 驱逐 regret 与缓存稳定性 |
| 长上下文压力 | 低到中 | 少量共享 | 2-8 | 显存压力、回收频率、重算成本 |

其中，前四类负载更适合验证本文提出的生命周期管理思路，第五类负载则用于确认策略在显存压力升高时是否仍然稳定。

### 11.3 对比方案设计

实验中可设置以下对比组：

- 原始 vLLM 默认机制；
- 仅启用前缀缓存的机制；
- 引入生命周期统计但不改变驱逐策略的机制；
- 引入生命周期感知协同驱逐策略的机制；
- 若实现条件允许，再加入局部分叉共享增强机制。

如果实验时间允许，对比组还应包含“prefix caching 禁用/启用”这一本底对照。这样可以把“前缀共享本身带来的收益”和“在已有共享基础上进一步做生命周期协同优化带来的增益”区分开来，避免把不同层次的效果混在一起解释。

### 11.4 评估指标

实验拟重点观察以下指标：

1. **前缀复用命中率**  
   衡量系统成功复用已有前缀 block 的程度。

2. **平均请求延迟**  
   衡量整体服务响应性能。

3. **首 token 时延（TTFT）**  
   对交互式服务尤其关键。

4. **吞吐率**  
   衡量单位时间可处理的请求或 token 数。

5. **显存占用情况**  
   包括平均显存使用量和高峰显存占用。

6. **Block 利用率与驱逐频率**  
   衡量缓存分配与回收是否高效。

7. **驱逐后的重算代价**  
   衡量回收策略是否导致过多重复 prefill。

为了保证评估结果可解释，部分指标还需要进一步形式化。前缀 token 命中率可定义为“被直接复用的 prompt token 数 / 可复用 prompt token 总数”，比单纯统计 request hit/miss 更细；block 复用率可定义为“命中已有 full block 数 / 本次请求需要的 full block 数”，更贴近 block 级机制；驱逐 regret 可定义为“在时间窗口 $W$ 内被驱逐后又被重建的 block 数 / 被驱逐 block 总数”，用于衡量错误淘汰；重算比例可定义为“由于未命中或驱逐导致重新 prefill 的 token 数 / 总 prefill token 数”；有效 block 利用率则可定义为“实际承载有效 token 的 block 容量 / 已分配 block 总容量”，用于观察碎片与尾块浪费。

同时，在延迟类指标上，除了平均值，还应记录 p50 / p95 TTFT 与端到端延迟。因为缓存策略对系统的影响常常首先体现为尾延迟波动，而不是平均值变化。

### 11.5 实验结果预期

预期在以下方面观察到收益：

- 对于共享前缀明显的请求集，前缀命中率提高；
- 对于局部分叉场景，共享能力优于只支持完全前缀匹配的基线；
- 生命周期感知驱逐能够减少“驱逐后很快又要重算”的无效淘汰；
- 在相同显存条件下，系统的整体服务效率更稳定。

需要强调的是，本项目的“预期收益”并不是所有指标都单调改善。举例来说，引入更多元数据和二阶段驱逐判断，可能会带来极小的管理开销；真正有价值的结果是：在相同显存预算下，总体重算下降、TTFT 稳定性提升、prefix hit 更持久，而不是要求每一个 microbenchmark 都绝对更快。因此，实验解释应更关注“收益来源是否与设计假设一致”。

### 11.6 消融实验与结果解释框架

为验证各个因子的独立贡献，本项目可设计以下消融实验：

1. 去掉共享度因子，只保留热度与 LRU，对比热点前缀是否更容易被误驱逐；
2. 去掉重算代价因子，观察长前缀场景下的重建开销是否明显上升；
3. 去掉前缀位置因子，观察“前缀前段 block”与“尾部 block”是否被一视同仁，从而损害后续复用；
4. 仅做生命周期统计而不改驱逐策略，验证 instrumentation 本身不会造成显著性能偏移。

通过消融实验，可以把“策略整体有效”进一步拆解为“哪些因素真正提供了收益”，这会使最终报告更具说服力。

## 12. 项目可行性分析

本章从理论依据、技术依据、工程条件、实验条件和风险控制等方面，对项目可行性进行集中分析。

### 12.1 理论依据与理论可行性

本项目的问题定义明确：分配、复用与驱逐存在天然耦合，统一生命周期建模具有充分理论动机。

### 12.2 技术依据与技术可行性

vLLM 已提供块式分配、前缀缓存与回收路径，且现有 `KVCacheBlock`、`BlockPool` 与 prefix 统计接口可直接复用。项目因此可采用增量改造路线，技术风险主要在策略组织而非底层可实现性。

### 12.3 工程可行性

工程条件明确：本地平台支持高频迭代，模型规模与上下文长度可控，能够支撑源码分析、原型实现与 A/B 对比。

### 12.4 实验可行性

实验目标以机制验证和相对比较为主，可通过典型负载观察命中、驱逐与重算差异；再在 RTX 3090 上补充关键复跑，符合分层验证路径，具备可执行性。

### 12.5 风险可控性

主要风险包括源码复杂度、共享增强实现难度和显存边界；通过分层目标、增量实现和服务器补充验证，整体风险可控。

### 12.6 阶段性成功标准

为了避免把项目成败绑定在“必须完成全部高级共享机制”上，本文将阶段性成功标准分为三个层次：

1. **基础成功**：完成 vLLM 基线复现、源码路径梳理、生命周期统计实现，并给出 prefix hit、eviction、rebuild 的可复现实验数据；
2. **中级成功**：在此基础上实现生命周期感知驱逐，并在至少一种共享前缀负载上获得更低的重算比例或更稳定的 TTFT；
3. **完整成功**：进一步实现共享后分叉的结构化管理，在局部分叉或逐步分叉负载上取得优于默认机制的结果。

这种分层目标设定使项目具备很强的可控性：即使高级共享机制没有完全落地，项目也仍然能够形成完整、可信的研究闭环。

## 13. 可能的风险与应对措施

### 13.1 显存资源有限

当前设备仅有 8GB 显存，限制了模型规模和实验负载。对此可采取以下策略：

- 选用 2B 级别小模型，并限制为 text-only 运行；
- 限制 `max_model_len`；
- 聚焦机制验证而非大模型绝对性能；
- 优先观察命中率、驱逐行为和显存利用趋势。

### 13.2 vLLM 源码复杂度较高

vLLM 涉及 engine、scheduler、worker、distributed 和 cache manager 等多个模块。应对方式是：

- 先从关键调用链入手；
- 重点阅读与 KVCache 直接相关的模块；
- 采用“从入口到关键机制”的增量阅读方式，而不是全面铺开。

### 13.3 共享增强机制实现难度较高

近似前缀匹配、chunk 级共享和写时复制等机制实现复杂，可能难以在短周期内全部完成。对此可以：

- 先实现最小版本的生命周期统计；
- 再优先改造驱逐策略；
- 共享增强部分以设计与局部原型为主，避免过度追求完整系统实现。

### 13.4 性能收益不显著或结论不稳定

系统策略类项目常见风险是不出现明显“绝对提速”，或者不同负载下结论不一致。对此应提前调整评价方式：不把单一吞吐提升作为唯一成功标准，而是同时观察 prefix 命中率、驱逐 regret、重算比例、TTFT 波动和显存利用率；在结果解释上，更强调“哪些场景受益、哪些场景无显著收益、原因是什么”。只要能够清晰说明生命周期建模在何种条件下有效，本项目仍然具有研究价值。

## 15. 结论与预期成果

本文围绕“面向 LLM 推理服务的 KVCache 分配、复用与驱逐协同优化”这一主题，对项目的研究背景、问题定义、核心思想、技术路线、实现基础、实验设计和可行性进行了系统分析。综合来看，本项目具有以下特点：

1. 研究问题具有现实意义，直接面向 LLM 在线服务中的关键瓶颈；
2. 项目关注点明确，围绕 KVCache 分配、共享和驱逐的耦合关系展开；
3. 核心思想具有一定创新性，即从统一生命周期视角建模 KV block；
4. 已有开源系统和实验环境为项目实施提供了充分基础；
5. 即使在有限硬件条件下，项目仍具备开展机制验证与原型实验的可行性。

因此，可以认为本项目具有较强的理论价值和工程可操作性，适合作为后续研究和实践的重点方向。

本项目预期将形成以下成果：

- 对现有 KVCache 管理机制的系统性分析；
- 一个统一的 KVCache 生命周期管理框架；
- 一种协同优化的分配—复用—驱逐策略设计；
- 基于 vLLM 的原型验证或半实现验证；
- 一套可复现的工作负载构造与指标采集脚本；
- 对 vLLM 关键模块切入点和改造边界的实现分析；
- 一组针对典型请求模式的实验结果与分析结论；
- 一份完整的项目研究报告。

未来，如果条件允许，项目还可进一步扩展到更复杂的近似匹配、跨请求 chunk 级共享和更精细的服务级调度协同优化，为高性能 LLM Serving 系统提供更系统的 KVCache 管理思路。
